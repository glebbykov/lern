# 🛡️ Project Aegis: Distributed Zero-Trust Infrastructure

## 📖 Легенда (Business Case)
Вы — Lead Infrastructure Engineer в стартапе **"Aegis Pay"**. 
По требованию архитектурного комитета и с учетом квот облачного провайдера (ограничение на vCPU), инфраструктура должна быть **физически разделена на изолированные узлы** по типу нагрузки. 

Мы заменяем legacy-компоненты на современные решения: вместо HAProxy мы используем **Nginx**, а вместо RabbitMQ мы разворачиваем **Apache Kafka** в новейшем режиме **KRaft** (без Zookeeper).

## 🏗 Архитектура и Топология Кластера (Best Practices)

Для имитации серьезного Enterprise Data-Center мы разворачиваем 4 изолированные виртуальные машины (каждая ограничена по vCPU для вписывания в жесткие квоты облака, но обладает всеми свойствами больших серверов):

1. **`bastion` (DMZ):** Единственная точка входа по SSH (белый список IP).
2. **`app-node` (Ingress & Compute):** Здесь работает `Nginx` (как L7 Ingress-балансировщик) и `Containerd` в Systemd-песочнице.
3. **`db-node` (Relational & Cache):** Здесь "живут" `PostgreSQL` и `Redis`. К серверу прикреплены 2 диска, собранные в **RAID 1 (Зеркало)**.
4. **`kafka-node` (Message Broker):** Выделенный сервер под `Apache Kafka` (в режиме KRaft). К нему прикреплены 3 диска, собранные в **RAID 5 (Чередование с четностью)**, а хранилище отформатировано в **XFS**, что критически важно для производительности Kafka.

### 🔥 Продвинутый Тюнинг Баз Данных (Stateful Tier)
Поскольку базы разнесены, мы применяем гранулярный тюнинг ядра (через Ansible) на каждом узле:

- **App Node (Nginx):** 
  - Настройка **TCP BBR** (Bottleneck Bandwidth and RTT) для ускорения отдачи трафика.
  - Расширение эфемерных портов (`net.ipv4.ip_local_port_range`) для предотвращения исчерпания соединений (TCP Port Exhaustion) при балансировке.
- **DB Node (PostgreSQL + Redis):**
  - Отключение **Transparent Huge Pages (THP)** через systemd-oneshot юнит (Избавляет Redis от latency-спайков).
  - Настройка `vm.dirty_background_ratio = 5` и `vm.dirty_ratio = 15`. Заставляет ядро плавно сбрасывать данные (Dirty Pages) Postgres на диск, избегая зависаний I/O.
- **Kafka Node:**
  - Kafka невероятно сильно полагается на **Page Cache** (остаток RAM, используемый ядром для кэширования диска).
  - Настраиваем `vm.dirty_ratio = 60` — даем Кафке возможность держать больше данных в RAM перед сбросом на RAID 5.
  - `vm.max_map_count = 262144` — предотвращаем OOM при создании тысяч партиций.

## 📋 Инструкция по развертыванию

1. **Provisioning (Создание Облака):**
   Выполните скрипт:
   ```bash
   ./provision/azure_create.sh
   ```
   Он создаст 4 виртуалки (включая разметку пустых дисков для RAID-массивов) и выведет настройки для `~/.ssh/config`.

2. **Ansible Automation:**
   Перейдите в директорию `ansible` и запустите плейбук.
   ```bash
   ansible-playbook -i inventory/hosts.ini site.yml
   ```
   *Ansible автоматически определит, на каком сервере он находится, соберет нужный RAID, применит нужный sysctl.d тюнинг, развернет Nginx, PostgreSQL, Redis, скачает Kafka, сгенерирует KRaft-кластер и настроит UFW/Auditd.*

Это финальная точка кристаллизации ваших знаний: от сетей и дисков до тюнинга ядра и автоматизации. Идеальный проект для GitHub и резюме!
