# 🛡️ Project Aegis: Zero-Trust Production Infrastructure

## 📖 Легенда (Business Case)
Вы — Lead Infrastructure Engineer в перспективном FinTech-стартапе **"Aegis Pay"**. Компания готовится к прохождению аудита безопасности (PCI-DSS) и глобальному релизу. 
Исторически инфраструктура разворачивалась вручную: приложения работают от пользователя `root`, порты торчат наружу, используются дефолтные параметры ядра (из-за чего при нагрузке падают TCP-соединения), а базы данных хранятся на одном системном диске вместе с ОС.

**Ваша миссия:** С нуля спроектировать и развернуть отказоустойчивую, защищенную по принципам **Zero-Trust** и **Defense-in-Depth** инфраструктуру. Эта песочница должна стать идеальным фундаментом для будущего развертывания Kubernetes кластера (Kubeadm/K3s) и микросервисов. Вся конфигурация должна управляться через концепцию Infrastructure as Code (Ansible/Terraform/Bash).

## 🏗 Теоретическое обоснование и Лучшие Практики (Best Practices)

Проект строится на индустриальных стандартах и решает конкретные инженерные задачи:

### 1. Архитектура сети (Network Partitioning & Bastion)
**Best Practice:** Серверы приложений и баз данных никогда не должны иметь публичных IP-адресов.
Мы разделяем виртуальную сеть (VNet) на `DMZ` (демилитаризованная зона) и `PrivateApp`. 
В DMZ находится единственный узел — `bastion-host`. Доступ к нему ограничен по IP администратора через Network Security Group (NSG). Все внутренние серверы доступны только через Бастион (используя SSH `ProxyJump`).

### 2. Дисковая подсистема (Storage Engineering)
**Best Practice:** Разделение I/O потоков и изоляция точек монтирования.
Хранить данные СУБД и контейнеров на системном диске (`/`) — это путь к катастрофе (IOPS голодание, исчерпание места приводит к Kernel Panic). Мы используем **LVM (Logical Volume Manager)** для гибкого управления дисками (позволяет расширять разделы "на горячую").
- **`etcd` (Хранилище стейта K8s):** Выделяется в отдельный логический том. Алгоритм Raft крайне чувствителен к задержкам синхронизации на диск (`fsync latency`). Соседи по диску не должны влиять на etcd.
- **`PostgreSQL`:** Отдельный том, оптимизированный под транзакционную нагрузку.
- **`containerd`:** Том форматируется в файловую систему `XFS` с флагом `pquota` (Project Quotas). Это фундаментальное требование Kubernetes для работы драйвера `overlayfs`, позволяющее ограничивать размер диска для отдельных контейнеров (ephemeral storage limits).

### 3. Тюнинг Ядра (Kernel Tuning)
**Best Practice:** Дефолтное ядро Linux рассчитано на десктопы/слабые серверы, а не на HighLoad.
- `swapoff`: Kubernetes требует полного отключения Swap, чтобы планировщик (kube-scheduler) мог гарантировать качество обслуживания (QoS) по памяти.
- `fs.inotify.max_user_watches`: Увеличивается для корректной работы систем, следящих за изменениями тысяч файлов (важно для CI/CD и баз данных).
- `net.core.somaxconn` и `net.ipv4.tcp_max_syn_backlog`: Увеличение очередей соединений (защита от SYN-флуда и дропов при всплесках трафика).

### 4. Харденинг и Изоляция (Security & Sandboxing)
**Best Practice:** Принцип наименьших привилегий (PoLP) на всех уровнях ОС.
- **Firewall:** UFW настраивается в режиме `Default Deny`. Разрешены только легитимные порты и только из доверенных подсетей.
- **Auditd:** Аудит всех попыток модификации `/etc/shadow`, конфигураций ядра и логов.
- **Systemd Sandboxing:** Любые не-контейнеризованные демоны запускаются в жестких песочницах systemd (запрет доступа к `/home`, read-only файловая система, отключение возможности эскалации привилегий `NoNewPrivileges`).

## 📋 План реализации (Эпики и Задачи)

### Epic 1: Cloud Provisioning (Инфраструктура)
- [ ] Написать скрипт `provision/azure_create.sh` для развертывания:
  - Resource Group `rg-aegis-prod`.
  - VNet `10.10.0.0/16`.
  - Подсеть DMZ (`10.10.1.0/24`) и Подсеть App (`10.10.2.0/24`).
  - Узел `bastion` (DMZ, Public IP, NSG с ограничением по вашему белому IP).
  - Узел `app-node` (App Subnet, без Public IP, +2 пустых Data-диска по 20GB).
- [ ] Настроить локальный `~/.ssh/config` для прозрачного прыжка на `app-node`.

### Epic 2: Ansible Foundation & Storage
- [ ] Создать базовый inventory файл для Ansible.
- [ ] Написать роль `01-storage`, которая:
  - Инициализирует Physical Volumes (PV) из двух прикрепленных дисков.
  - Создает Volume Group (VG) `vg_aegis_data`.
  - Создает логические тома (LV): `lv_etcd` (5GB), `lv_pgsql` (15GB), `lv_containerd` (15GB).
  - Форматирует их (XFS с `pquota` для containerd, ext4 для остальных) и монтирует в `/etc/fstab`.

### Epic 3: Kernel & Network Optimization
- [ ] Написать роль `02-kernel`, которая:
  - Отключает swap в системе и в `/etc/fstab`.
  - Загружает модули ядра `overlay` и `br_netfilter`.
  - Раскатывает конфиг `/etc/sysctl.d/99-aegis-prod.conf` (ip_forward, tcp_tw_reuse, inotify watches, file-max) и делает `sysctl --system`.

### Epic 4: Hardening & Compliance
- [ ] Написать роль `03-security`, которая:
  - Устанавливает и настраивает `ufw` (Разрешить SSH только с IP бастиона, запретить всё остальное).
  - Настраивает `sshd_config` (отключить root login, отключить пароли).
  - Устанавливает `auditd` и копирует правила для слежения за критическими путями.

### Epic 5: Runtime & Service Sandboxing
- [ ] Написать роль `04-runtime`:
  - Установка пакетов `containerd.io` и `runc`.
  - Конфигурация `containerd` для использования `SystemdCgroup = true`.
- [ ] Написать роль `05-app-sandbox`:
  - Установка тестового веб-приложения (например, на Python).
  - Развертывание systemd-unit файла с максимальными ограничениями: `ProtectSystem=strict`, `ProtectHome=yes`, `PrivateTmp=yes`, `NoNewPrivileges=yes`, `SystemCallFilter=~@mount @clock`.
