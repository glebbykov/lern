# 🛡️ Project Aegis: Zero-Trust Bare-Metal & Stateful Infrastructure

## 📖 Легенда (Business Case)
Вы — Lead Infrastructure Engineer в перспективном FinTech-стартапе **"Aegis Pay"**. 
Вам поручено спроектировать **Production-Grade Bare-Metal** песочницу. В этой инфраструктуре должен работать как будущий Kubernetes-кластер, так и выделенный **Слой данных (Stateful Tier)**, который по требованиям безопасности и производительности не должен деплоиться внутрь Kubernetes.

**Ваша миссия:** Автоматизировать развертывание сложной инфраструктуры, объединяющей балансировщики, базы данных, очереди сообщений и OCI-рантаймы с помощью **Ansible** и **Azure CLI**.

## 🏗 Архитектура и Лучшие Практики (Best Practices)

### 1. Почему мы выносим Базы Данных из Kubernetes? (Stateful Tier)
В средних и крупных проектах реляционные БД и кэши часто выносятся из K8s на классические VM. 
*Причины:*
- **Изоляция ресурсов (Noisy Neighbors):** Базы данных требуют монопольного доступа к кэшам процессора и оперативной памяти. В K8s планировщик может посадить рядом "шумный" под, который вызовет нестабильные задержки (latency spikes) в базе.
- **Тюнинг ядра:** Базы данных требуют жесткого изменения параметров ядра (например, отключение Transparent Huge Pages для Redis), что невозможно или крайне трудно сделать в изоляции для конкретного пода в K8s.
- **Storage I/O:** Kubernetes CSI драйверы добавляют слой абстракции, который замедляет работу дисковой подсистемы.

### 2. "Зоопарк" Баз Данных и их Тюнинг (Стек проекта)
Мы разворачиваем 4 классических Stateful-компонента и для каждого делаем индустриальный тюнинг ОС:

1. **PostgreSQL (Главная реляционная БД):**
   - *Проблема:* Ядро Linux по умолчанию кэширует слишком много данных в RAM перед сбросом их на диск, что вызывает огромные "лаги" I/O во время чекпоинтов.
   - *Тюнинг:* Настраиваем `vm.dirty_background_ratio = 5` и `vm.dirty_ratio = 15`. Ядро будет плавно и чаще сбрасывать данные на диск в фоне, избегая резких тормозов.
2. **Redis (In-Memory Кэш и Сессии):**
   - *Проблема:* Redis делает бэкапы (BGSAVE) через механизм `fork()`. Если памяти мало, ядро убьет процесс. Кроме того, механизм ядра Transparent Huge Pages (THP) вызывает чудовищные утечки памяти и тормоза в Redis.
   - *Тюнинг:* `vm.overcommit_memory = 1` (позволяет форкаться без проверки лимитов) и создание отдельного `systemd` демона, который отключает `THP` на уровне ядра при старте сервера.
3. **RabbitMQ (Брокер очередей сообщений):**
   - *Проблема:* Написан на Erlang, поддерживает тысячи одновременных коннектов. Дефолтный лимит Linux убьет его ошибкой `Too many open files`.
   - *Тюнинг:* `fs.file-max = 2000000` (Глобальный лимит дескрипторов).
4. **HAProxy (Load Balancer / Ingress Gateway):**
   - *Проблема:* При балансировке тысяч запросов в секунду серверу не хватает эфемерных портов для создания исходящих соединений к бэкенду.
   - *Тюнинг:* Расширяем `net.ipv4.ip_local_port_range = 1024 65535` и включаем алгоритм **TCP BBR** (Bottleneck Bandwidth and RTT) для ускорения отдачи трафика клиентам.

### 3. Архитектура Storage (RAID + LVM + XFS)
Мы прикрепляем 5 сырых дисков и объединяем их в 2 RAID-массива:
1. **RAID 1 (Зеркало, `vg_fast`):** Диски `sdc` и `sdd`. Отдаем под `etcd` (K8s State) и `Redis`. Супер-быстрая запись, не страдает при потере 1 диска.
2. **RAID 5 (Чередование с четностью, `vg_data`):** Диски `sde`, `sdf`, `sdg`. Баланс скорости и отказоустойчивости. Отдаем под `PostgreSQL`, `RabbitMQ` и `Containerd` (где XFS отформатирован с поддержкой `pquota`).

### 4. Харденинг и Изоляция
- **UFW (Default Deny):** Блокирует всё. Открыты только порты для внутреннего общения БД. Публичного IP нет вообще. Доступ админа — строго через отдельный `bastion-host`.
- **Systemd Sandboxing:** Кастомные приложения (наш Python веб-сервер) работают от `aegis-svc`, файловая система перемонтируется в Read-Only, блокируется эскалация привилегий и отрезаются системные вызовы через Seccomp.

## 📋 Инструкция по развертыванию

1. **Создание инфраструктуры в Azure:**
   Запустите скрипт `provision/azure_create.sh`. Он развернет Бастион и мощную App-ноду (`Standard_B2ms` - 8GB RAM) с пятью дисками.
2. **Запуск Ansible:**
   Отредактируйте `ansible/site.yml` (раскомментируйте новую роль `06-stateful-tier`).
   ```bash
   cd ansible
   ansible-playbook -i inventory/hosts.ini site.yml
   ```
