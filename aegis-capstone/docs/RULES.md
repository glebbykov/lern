# Правила документации Aegis Capstone

Этот файл — **мета-документ**. Описывает, как писать всё остальное.

Цель документации: чтобы и человек, и LLM-агент могли за минуту понять «где правда, где компромисс, и как воспроизвести».

---

## 1. Шесть принципов

### 1.1 Single Source of Truth (SSOT)
Каждый факт живёт **в одном месте**. Дубли запрещены — они расходятся, и ты теряешь доверие к доке.

- IP-адреса → `terraform output`.
- Порты → [`docs/ports.md`](ports.md).
- Версии провайдеров → `terraform/versions.tf`.
- Имена групп Ansible → `ansible/inventory/hosts.ini`.

Если хочется упомянуть факт в другом месте — **поставь ссылку**, не копию.

### 1.2 Machine-readable + human-readable layered
Каждый значимый `.md` устроен слоями:
1. **YAML frontmatter** — для ботов и индексации.
2. **TL;DR / Context** — для человека за 30 секунд.
3. **Подробности с примерами** — для того, кто реально работает.
4. **Mermaid-диаграммы** — для обоих.

### 1.3 WHY > WHAT
Код уже говорит **что делает**. Дока говорит **почему именно так**, какие были альтернативы, какие trade-offs.
Если в комментарии или абзаце нет ответа на «почему» — либо переписать, либо удалить.

### 1.4 Diagrams as code, не PNG
Только Mermaid / D2 / PlantUML внутри `.md`. PNG/JPEG не коммитим:
- GitHub рендерит Mermaid нативно.
- Diff в git осмысленный.
- LLM читает текст.

Исключение — реальные скриншоты Grafana/UI (тогда в `docs/img/` с осмысленным именем).

### 1.5 Stable anchors
Имена файлов, заголовки H1/H2 и якоря — стабильны. Если переименовываешь — оставь редирект-стаб:
```markdown
> Этот документ переехал в [новое-имя.md](новое-имя.md).
```

### 1.6 Generated ≠ authored, и они не смешиваются
Файл либо **целиком сгенерирован** — тогда сверху баннер `<!-- DO NOT EDIT — generated by ... -->` — либо **целиком ручной**. Никаких «частично сгенерированных» — это всегда заканчивается затёртыми правками (см. [ADR-0005](adr/0005-remove-generate-tf-py.md)).

## 2. Frontmatter: обязательный заголовок

Каждый `.md` в `docs/` и в корне проекта начинается с:

```yaml
---
title: Короткое название документа
status: draft | stable | deprecated
audience: [ops, llm, contributors, reviewers]   # один или несколько
last_verified: YYYY-MM-DD                       # когда последний раз сверялся с реальностью
related:                                        # опционально
  - docs/adr/0003-disk-isolation.md
  - ansible/roles/06-stateful-tier/
---
```

**Зачем:**
- `status` — читатель сразу понимает, можно ли доверять.
- `last_verified` — после этой даты всё могло устареть; LLM это учтёт.
- `related` — машинно-читаемый граф связей.

`README.md` в корне репозитория — единственное исключение (его читают сторонние без контекста, frontmatter мешает).

## 3. Структура `docs/`

```
docs/
├── RULES.md            # этот файл
├── ports.md            # SSOT всех портов
├── topology.md         # SSOT сетевой топологии
├── glossary.md         # канонические термины
├── PROJECT_PLAN.md     # план приложения (что и в каком порядке строим)
├── adr/                # Architectural Decision Records
│   ├── README.md       # индекс ADR
│   ├── template.md     # шаблон для новых
│   ├── 0001-*.md
│   └── ...
└── runbooks/           # пошаговые операционные процедуры
    ├── deploy.md
    └── ...
```

**Правила размещения:**
- ADR — для **решений**, не для инструкций. «Почему мы выбрали PG, а не MySQL» = ADR.
- Runbook — для **действий**. «Как раскатать инфру с нуля» = runbook.
- Если документ описывает компонент — он живёт **рядом с кодом** (`ansible/roles/X/README.md`), а не в `docs/`.

## 4. Когда писать ADR

Пиши ADR, если решение:
- **Нетривиальное** — есть минимум одна разумная альтернатива.
- **Дорого откатить** — миграция БД, смена облака, разделение/объединение сервисов.
- **Влияет на других** — другие команды/сервисы зависят от него.

Не пиши ADR на `terraform fmt`, переименование переменной, починку опечатки.

**Формат ADR — Майкла Найгарда** (см. [adr/template.md](adr/template.md)). Минимум:
- Status, Context, Decision, Consequences (+/−), Alternatives considered.

ADR **не редактируются после Accepted** — если решение пересмотрено, создаётся новый ADR со `status: Accepted` и ссылкой `Supersedes: 000X`. Старый получает `status: Superseded by 000Y`.

## 5. Стиль текста

### Краткость
- TL;DR в начале любого документа > 30 строк.
- Абзац на одну мысль. Если в абзаце два «и поэтому» — режь.
- Списки лучше, чем proza, если перечисляются однородные сущности.

### Команды и пути
- Команды — в блоках кода с указанием контекста (`cd terraform && ...`), а не голым `terraform apply`.
- Пути — абсолютные относительно корня репо (`ansible/roles/01-storage/`), не `./roles/...`.
- Имена хостов — канонические (`az-app`, не «application server»).

### Ссылки
- Всегда относительные внутри репо: `[ADR-0003](adr/0003-disk-isolation.md)`.
- Внешние ссылки — с описанием, не голый URL: `[Mermaid syntax](https://mermaid.js.org/syntax/flowchart.html)`.

### Что НЕ писать
- Эмодзи в технических документах (исключение — README.md, маркетинговая страница).
- «Очевидно», «просто», «легко» — обнуляет доверие читателя, у которого не получилось.
- Capitalized Marketing Phrases типа «Enterprise-Grade Solution» в инженерных доках.

## 6. Inline-комментарии в коде

### Terraform
```hcl
# Standard SKU обязателен для статического PIP в Availability Zones.
# См. ADR-0004.
resource "azurerm_public_ip" "app" {
  ...
}
```

### Ansible
- `name:` уже играет роль документации. Дополнительный комментарий нужен только если `when:` неочевиден или если есть гонка/порядок.
- Объяснения «почему этот sysctl именно такой» — рядом с задачей, **со ссылкой на ADR**.

### Что не комментировать
- Что делает строка (это видно).
- Историю изменений («тут был баг с X, поправил» → это в commit message).
- TODO без owner'а и контекста (`# TODO fix this` — мусор).
