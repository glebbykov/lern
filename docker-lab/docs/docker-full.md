Часть 1. Docker в production

Длительность: ~15 минут

1.1 Зачем Docker в production

Прежде чем погружаться в механику контейнеров, давайте ответим на главный вопрос: зачем Docker используется в реальном production? Это не академическое упражнение — это инструмент, который решает конкретные бизнес-проблемы.

Консистентность окружений. Один и тот же Docker-образ работает одинаково на ноутбуке разработчика, в CI/CD пайплайне, на staging и в production. Навсегда исчезает проблема «у меня работает» — потому что образ содержит всё окружение: runtime, библиотеки, конфигурацию, зависимости. Если образ прошёл тесты — он будет работать точно так же в production.

Скорость поставки. Традиционный деплой: обновить пакеты на сервере, перезапустить сервис, молиться что ничего не сломается. Docker-деплой: docker pull myapp:1.2.0 && docker compose up -d. Rollback: docker compose up -d myapp:1.1.0. Весь цикл — секунды, а не часы. А с Kubernetes — ещё и автоматический rolling update с canary.

Изоляция и плотность. На одном хосте могут работать сотни микросервисов, каждый в своём контейнере с собственными зависимостями. Один сервис на Python 3.8, другой на Python 3.12, третий на Node.js 20 — никаких конфликтов. Каждый контейнер видит только своё окружение.

Масштабирование. Нужно больше capacity? Запускаем больше контейнеров. Kubernetes делает это автоматически через Horizontal Pod Autoscaler — при росте нагрузки поднимает новые реплики, при спаде — убирает.

1.2 Production workflow: от кода до деплоя

В production Docker-workflow выглядит как конвейер:

Code → Build → Test → Push → Deploy
(git push)  (docker build)  (CI pipeline)  (registry)  (K8s / Compose)

Ключевые принципы этого workflow:

Immutable artifacts. Образ собирается один раз и деплоится везде. Никаких изменений в runtime — то, что протестировано, ровно то и попадает в production. Никаких apt-get install на production-серверах.

Infrastructure as Code. Dockerfile, compose.yaml, Kubernetes-манифесты — всё лежит в git. Воспроизводимость, code review, полная история изменений. Любой инженер может восстановить environment с нуля.

Observability built-in. HEALTHCHECK прописан в образе. Resource limits заданы в Compose/K8s. Метрики собираются через cAdvisor → Prometheus → Grafana. Контейнер рождается наблюдаемым — это не опция, которую «потом добавим».

1.3 Docker в реальной инфраструктуре

В реальной инфраструктуре Docker-образ проходит три зоны:

Developer. Разработчик пишет Dockerfile и compose.yaml. Локально запускает docker compose up для полного dev-стенда: приложение + БД + кеш + мониторинг. Весь стек поднимается одной командой, одинаково у каждого в команде.

CI/CD. При push в git CI-система (GitHub Actions, GitLab CI, Jenkins) автоматически собирает образ, прогоняет тесты внутри контейнера, сканирует на CVE через Trivy, и если всё зелёное — пушит образ в registry с semver-тегом и digest.

Production. Kubernetes (или Docker Compose для single-host) делает pull образа по digest (не по тегу!), запускает контейнеры с health checks, resource limits, и подключает monitoring. При проблемах — автоматический restart или rollback.

Docker-образ — это единый артефакт, который без изменений проходит весь путь от разработки до production.



Часть 2. Фундамент контейнеризации

Длительность: ~45 минут

2.1 Зачем нужны контейнеры: историческая перспектива

Чтобы по-настоящему понять Docker, нужно оглянуться назад и увидеть, какую проблему он решил. На протяжении десятилетий индустрия разработки ПО сталкивалась с одним и тем же фундаментальным вопросом: как гарантировать, что программа будет работать одинаково на разных машинах?

Эволюция изоляции

Эпоха bare metal (до 2000-х). Каждое приложение работало на выделенном физическом сервере. Это означало колоссальные затраты на оборудование, низкую утилизацию ресурсов (типично 10–15% загрузки CPU) и мучительно долгий цикл развёртывания. Чтобы запустить новый сервис, нужно было заказать сервер, дождаться доставки, настроить ОС — процесс, растягивавшийся на недели.

Эпоха виртуализации (2000–2013). VMware, Xen, KVM решили проблему утилизации: на одном физическом сервере появилось возможность запускать несколько виртуальных машин, каждая со своей ОС. Плотность выросла, но каждая VM по-прежнему тащила полноценное ядро, системные сервисы, загружалась минутами и потребляла сотни мегабайт RAM только на саму ОС.

Эпоха контейнеров (2013+). Docker не изобрёл контейнеризацию — технологии chroot (1979), FreeBSD Jails (2000), Solaris Zones (2004), Linux Containers (LXC, 2008) существовали задолго до него. Но Docker сделал то, что не удалось никому до него: создал удобный, стандартизированный пользовательский опыт вокруг контейнеров. Он упаковал сложные механизмы ядра Linux в простые команды — docker build, docker run, docker push — и предложил экосистему обмена образами через Docker Hub.

Проблема «у меня работает»

Классический кошмар разработки — фраза «у меня на машине всё работает». Причины этой проблемы:





Разные версии библиотек и runtime между dev/staging/prod



Системные зависимости, установленные вручную и забытые



Конфликты между приложениями, использующими разные версии одной библиотеки



Неявные зависимости от конфигурации ОС, переменных окружения, путей в файловой системе

Контейнер решает это радикально: он упаковывает приложение вместе со всеми его зависимостями в единый переносимый артефакт. Этот артефакт — Docker-образ — гарантированно запускается одинаково на любой машине, где есть Docker Engine.

1.2 Контейнеры vs виртуальные машины

Это один из самых частых вопросов на собеседованиях, и одновременно один из самых непонятых. Многие думают, что контейнер — это «лёгкая виртуальная машина». Это неверно и ведёт к ошибочным архитектурным решениям.

Виртуальная машина

VM — это полная эмуляция аппаратного обеспечения. Гипервизор (VMware ESXi, KVM, Hyper-V) предоставляет гостевой ОС виртуальный набор устройств: CPU, RAM, диск, сетевую карту. Внутри VM работает собственное ядро Linux (или Windows, или любой другой ОС), своя инициализация (systemd/init), свои драйверы. Между гостевой ОС и хост-ОС существует жёсткая граница — гипервизор.

Характеристики VM:





Полная изоляция на уровне ядра — разные ОС на одном хосте



Загрузка от 30 секунд до нескольких минут



Потребление RAM: сотни мегабайт только на ОС



Размер образа: гигабайты



Плотность: десятки VM на одном хосте

Контейнер

Контейнер — это изолированный процесс (или группа процессов), работающий на ядре хост-ОС. Контейнер не имеет собственного ядра, не загружает ОС, не эмулирует аппаратуру. Он использует механизмы ядра Linux — namespaces и cgroups — чтобы создать иллюзию отдельной системы для процесса внутри.

Характеристики контейнера:





Изоляция на уровне процесса — разделяет ядро с хостом



Запуск за миллисекунды (это просто запуск процесса)



Потребление RAM: только то, что нужно самому приложению



Размер образа: мегабайты — десятки мегабайт



Плотность: сотни и тысячи контейнеров на одном хосте

Ключевое отличие: уровень изоляции

VM изолируют на уровне аппаратуры (виртуальный CPU, RAM, диск). Контейнеры изолируют на уровне процессов (видимость файловой системы, сети, PID). Это фундаментальное различие определяет все остальные характеристики.

Следствия:





Контейнеры не безопаснее VM по умолчанию. Побег из контейнера (container escape) проще, чем побег из VM, потому что контейнер разделяет ядро с хостом. Эксплойт ядра = root на хосте.



Контейнеры быстрее VM, потому что не нужно загружать ОС.



Контейнеры легче VM, потому что не дублируют ядро.



Контейнеры работают только на Linux (Docker на macOS и Windows использует скрытую Linux VM).

1.3 Технологии ядра Linux: namespaces, cgroups, union filesystem

Docker — это надстройка над тремя фундаментальными механизмами ядра Linux. Без понимания этих механизмов невозможно по-настоящему понять, что происходит при запуске контейнера.

Linux Namespaces — изоляция видимости

Namespaces определяют, что видит процесс. Каждый namespace изолирует отдельный аспект системы:

PID namespace — изоляция дерева процессов. Процесс внутри контейнера видит себя как PID 1, хотя на хосте у него совершенно другой PID. Он не видит процессы других контейнеров и хоста.

NET namespace — изоляция сетевого стека. Каждый контейнер получает собственный набор сетевых интерфейсов, таблицу маршрутизации, правила iptables, сетевой стек. Контейнер «думает», что у него собственная сеть.

MNT namespace — изоляция точек монтирования. Контейнер видит свою файловую систему, отличную от файловой системы хоста. Это то, что позволяет иметь разное содержимое /etc, /usr, /lib в каждом контейнере.

UTS namespace — изоляция hostname и domain name. Каждый контейнер может иметь свой hostname.

IPC namespace — изоляция межпроцессного взаимодействия (shared memory, semaphores, message queues).

USER namespace — маппинг пользователей. Root внутри контейнера (uid 0) может быть mapped на непривилегированного пользователя на хосте. Это важный механизм безопасности, который, однако, не включён в Docker по умолчанию.

CGROUP namespace — изоляция представления cgroup-иерархии.

Когда Docker создаёт контейнер, он создаёт новые экземпляры каждого из этих namespaces и запускает процесс внутри них. Процесс живёт в «пузыре» из namespaces и считает себя единственным на машине.

Control Groups (cgroups) — ограничение ресурсов

Если namespaces отвечают за видимость (что процесс видит), то cgroups отвечают за ресурсы (сколько процесс может потребить). cgroups позволяют:

Ограничить CPU. Указать, что контейнер может использовать не более 50% одного ядра. Docker реализует это через параметры --cpus, --cpu-shares, --cpu-period, --cpu-quota.

Ограничить память. Указать жёсткий лимит RAM. При превышении ядро убивает процесс через OOM Killer — контейнер получает exit code 137 (128 + 9, SIGKILL).

Ограничить I/O. Управлять скоростью чтения/записи на диск через --device-read-bps, --device-write-bps.

Учитывать потребление. cgroups собирают статистику использования ресурсов каждым контейнером — именно эти данные показывает docker stats.

Без cgroups один «шумный сосед» — контейнер, который внезапно начал потреблять всю RAM или CPU — мог бы положить весь хост. В production среде ограничения ресурсов через cgroups — не опция, а обязательность.

Union Filesystem (OverlayFS) — слои образа

Union filesystem — это механизм, который позволяет «наслоить» несколько файловых систем друг на друга, создавая единое представление. Docker использует его для реализации системы слоёв.

Каждая инструкция в Dockerfile (RUN, COPY, ADD) создаёт новый read-only слой. Когда контейнер запускается, поверх всех read-only слоёв образа добавляется один writable слой — container layer. Все изменения файловой системы во время работы контейнера записываются в этот верхний слой.

Принцип Copy-on-Write (CoW): при первом изменении файла из нижнего слоя, файл копируется в верхний (writable) слой, и изменения записываются туда. Оригинал в нижнем слое остаётся нетронутым. Именно поэтому:





Образ можно разделять между множеством контейнеров (read-only слои общие)



Запуск контейнера практически мгновенный (не нужно копировать файловую систему)



При удалении контейнера теряется только writable слой



Размер образа — это сумма уникальных слоёв (общие слои считаются один раз)

В Docker на Linux по умолчанию используется overlay2 — наиболее зрелый и производительный union filesystem driver.

1.4 Архитектура Docker Engine

Docker Engine — это не одна монолитная программа, а набор компонентов, работающих вместе. Понимание архитектуры важно для диагностики и осознания того, что происходит при выполнении каждой команды.

Компоненты

Docker CLI (docker) — клиент командной строки. Когда вы вводите docker run, CLI формирует HTTP-запрос и отправляет его Docker Daemon через Unix-сокет (/var/run/docker.sock) или TCP.

Docker Daemon (dockerd) — главный сервер-демон. Принимает API-запросы, управляет образами, сетями, volumes. Но сам dockerd не запускает контейнеры напрямую. Он делегирует это containerd.

containerd — runtime высокого уровня. Управляет жизненным циклом контейнеров: pull образов, управление хранилищем, создание и мониторинг контейнеров. containerd — это отдельный проект CNCF, используемый не только Docker, но и Kubernetes (напрямую, без dockerd).

runc — runtime низкого уровня. Это реализация OCI Runtime Specification. runc выполняет конкретную работу: создаёт namespaces, настраивает cgroups, делает pivot_root и запускает процесс контейнера. После запуска runc завершается — контейнер работает как обычный процесс под управлением containerd-shim.

containerd-shim — прослойка между containerd и процессом контейнера. Позволяет containerd (и dockerd) перезапускаться без остановки работающих контейнеров. Каждый контейнер имеет свой shim-процесс.

Поток выполнения docker run

Когда вы вводите docker run nginx, происходит следующая цепочка:





Docker CLI парсит команду и отправляет HTTP POST запрос на /containers/create к dockerd через Unix-сокет.



dockerd проверяет, есть ли образ nginx локально. Если нет — делает pull из registry (по умолчанию Docker Hub).



dockerd передаёт запрос на создание контейнера containerd через gRPC.



containerd подготавливает bundle (rootfs + config) для контейнера.



containerd вызывает runc для создания и запуска контейнера.



runc создаёт namespaces, настраивает cgroups, монтирует overlay filesystem, выполняет pivot_root и запускает процесс (в данном случае nginx).



runc завершается. Контейнер работает под containerd-shim.



dockerd получает подтверждение от containerd и возвращает ответ CLI.

Весь этот процесс занимает миллисекунды. Понимание его критически важно при диагностике — если docker run зависает, нужно понимать, на каком этапе цепочки произошла проблема.

OCI (Open Container Initiative)

OCI — это открытый стандарт, определяющий два ключевых спецификации:





Image Specification — формат образа контейнера (слои, манифест, конфиг)



Runtime Specification — как запускать контейнер (config.json для runc)

Благодаря OCI, образы, собранные Docker, могут запускаться в Podman, containerd, CRI-O и любом другом OCI-совместимом runtime. Это убирает vendor lock-in.

1.5 Ментальная модель: image, container, layer, registry

Прежде чем двигаться дальше, зафиксируем ключевые концепции и их взаимосвязи.

Image (образ)

Образ — это read-only шаблон, содержащий файловую систему и метаданные для запуска контейнера. Образ состоит из упорядоченного набора слоёв. Каждый слой — это diff (набор изменений файловой системы) относительно предыдущего слоя.

Образ идентифицируется тремя способами:





Имя + тег: nginx:1.27-alpine (человекочитаемый, но mutable — тег можно перезаписать)



Digest: nginx@sha256:a1b2c3... (content-addressable, immutable — единственный надёжный способ идентификации)



Image ID: короткий хеш для локальных операций

В production всегда ориентируйтесь на digest, а не на теги. Тег latest — не «последняя версия», а просто тег по умолчанию, который авторы образа могут (или не могут) обновлять.

Container (контейнер)

Контейнер — это запущенный экземпляр образа. По аналогии с ООП: образ — это класс, контейнер — это объект (instance). Из одного образа можно создать сколько угодно контейнеров.

Контейнер = read-only слои образа + один writable слой сверху. Всё, что процесс записывает в файловую систему контейнера, попадает в этот writable слой и теряется при удалении контейнера (если не используются volumes).

Layer (слой)

Слой — это минимальная единица файловой системы образа. Каждая инструкция Dockerfile создаёт новый слой. Слои кешируются, дедуплицируются и разделяются между образами.

Если два образа имеют общий базовый образ (например, python:3.12-slim), их общие слои хранятся на диске один раз. Это экономит и дисковое пространство, и время при pull (скачиваются только отсутствующие слои).

Registry (реестр)

Registry — это хранилище образов. Docker Hub — публичный registry по умолчанию. Существуют также: GitHub Container Registry (ghcr.io), Google Artifact Registry, AWS ECR, Harbor (self-hosted) и другие.

Registry хранит образы, индексированные по имени и тегу, и отдаёт их по протоколу Docker Registry HTTP API V2. При docker pull, клиент скачивает манифест (список слоёв), а затем скачивает каждый слой по его content digest.



Часть 2. Жизненный цикл контейнера и CLI

Длительность: ~45 минут

2.1 Состояния контейнера: от created до removed

Контейнер — это процесс с жизненным циклом. Каждый контейнер проходит через чётко определённые состояния:

docker run
    │
    ▼
 created ──► running ──► paused
                │              │
                │              ▼
                │           resumed
                │
                ▼
             stopped ──► removed
                │
                ▼
           restarting  (по restart policy)

created — контейнер создан, но процесс ещё не запущен. Это состояние можно увидеть при docker create (без docker start).

running — основной процесс контейнера (PID 1) работает.

paused — процесс заморожен через cgroup freezer. Контейнер не потребляет CPU, но RAM остаётся занятой. Используется редко, но полезно для отладки.

stopped — процесс завершился (сам или по сигналу). Контейнер и его writable слой всё ещё существуют на диске. Можно посмотреть логи, inspect, перезапустить.

removed — контейнер полностью удалён: writable слой очищен, метаданные стёрты. Необратимая операция.

restarting — Docker перезапускает контейнер согласно restart policy (если настроена).

2.2 PID 1 внутри контейнера: почему это критически важно

Внутри контейнера первый запущенный процесс получает PID 1. В обычной Linux-системе PID 1 — это init (systemd), который выполняет две специальные функции:





Обработка сигналов. PID 1 — это единственный процесс, которому Docker посылает сигналы (SIGTERM при docker stop, SIGKILL при docker kill). Если PID 1 не обрабатывает SIGTERM, процесс не завершится gracefully, и через 10 секунд Docker пошлёт SIGKILL.



Сбор зомби-процессов (zombie reaping). Когда дочерний процесс завершается, он становится «зомби» — запись в таблице процессов, ожидающая, пока родитель вызовет wait(). В обычной системе это делает init. Если PID 1 в контейнере — ваше приложение, а оно не вызывает wait() для дочерних процессов, зомби накапливаются.

Типичная проблема: shell как PID 1

Рассмотрим Dockerfile:

CMD ["sh", "-c", "python app.py"]

Здесь PID 1 — это sh. При docker stop SIGTERM получает sh, но sh по умолчанию не форвардит сигнал дочернему процессу python. Python продолжает работать, пока через 10 секунд Docker не пришлёт SIGKILL всем. Приложение не успевает закрыть соединения, дописать данные, завершить транзакции.

Решения

exec form в CMD/ENTRYPOINT — запускает процесс напрямую, без shell:

CMD ["python", "app.py"]   # exec form — python становится PID 1

exec в shell-скрипте — заменяет shell на приложение:

#!/bin/sh
exec python app.py    # exec заменяет shell → Python становится PID 1

dumb-init или tini — специальный init-процесс для контейнеров:

RUN pip install dumb-init
ENTRYPOINT ["dumb-init", "--"]
CMD ["python", "app.py"]

dumb-init (или tini, встроенный в Docker через --init) берёт на себя обе обязанности PID 1: форвардит сигналы и собирает зомби.

2.3 Сигналы, коды выхода и graceful shutdown

docker stop vs docker kill

docker stop <container>        # SIGTERM → ждёт 10s → SIGKILL
docker stop -t 30 <container>  # SIGTERM → ждёт 30s → SIGKILL
docker kill <container>        # Немедленный SIGKILL
docker kill -s SIGHUP <ctr>    # Послать конкретный сигнал

docker stop — правильный способ для production. Он даёт приложению время на graceful shutdown: закрыть соединения с БД, завершить текущие запросы, сохранить состояние.

docker kill — когда контейнер завис и не реагирует на SIGTERM. Это экстренная мера.

Коды выхода (Exit Codes)

Код выхода — это число от 0 до 255, которое возвращает процесс при завершении. По нему можно понять причину остановки контейнера:







Код



Значение





0



Нормальное завершение





1



Общая ошибка приложения





125



Ошибка самого Docker daemon (неверные флаги команды)





126



Команда найдена, но не может быть выполнена (нет прав)





127



Команда не найдена (опечатка в CMD/ENTRYPOINT)





130



Завершение по SIGINT (Ctrl+C), 128 + 2





137



Завершение по SIGKILL (OOMKill или docker kill), 128 + 9





143



Завершение по SIGTERM (docker stop, graceful), 128 + 15

Формула: 128 + номер сигнала = exit code. SIGKILL = 9, SIGTERM = 15.

Диагностика:

docker inspect <ctr> --format '{{.State.ExitCode}}'
docker inspect <ctr> --format '{{.State.OOMKilled}}'

Если exit code = 137 и OOMKilled = true — контейнер убит по превышению лимита памяти. Если 137 и OOMKilled = false — кто-то (или Docker) послал SIGKILL вручную.

2.4 Restart policies и их подводные камни

Docker может автоматически перезапускать упавшие контейнеры:







Policy



Поведение





no (default)



Не перезапускать никогда





always



Перезапускать всегда, включая после reboot хоста





unless-stopped



Перезапускать, кроме ручной остановки





on-failure[:N]



Только при ненулевом exit code, максимум N раз

Критическая ловушка always: если контейнер падает из-за ошибки конфигурации (неверный путь к файлу, недоступная БД, невалидный конфиг), restart: always создаёт бесконечный crashloop — контейнер падает, перезапускается, снова падает. Это генерирует огромное количество логов, потребляет ресурсы и маскирует реальную проблему.

Рекомендации:





Для production-сервисов: unless-stopped или on-failure:5



Никогда не используйте always как замену исправлению реальной причины падения



Проверяйте docker events для отслеживания циклов перезапуска

# Изменить политику у работающего контейнера
docker update --restart unless-stopped <container>

2.5 Диагностика: logs, inspect, events, stats

Набор инструментов для диагностики контейнеров — это ваш ежедневный инструмент. Каждая из этих команд отвечает на свой вопрос:

docker logs — «что приложение выводит?»

docker logs <ctr>              # все логи
docker logs <ctr> --tail 50    # последние 50 строк
docker logs <ctr> -f            # потоковый вывод (follow)
docker logs <ctr> --since 10m   # за последние 10 минут
docker logs <ctr> -t            # с timestamps

Docker перехватывает stdout и stderr процесса контейнера. Именно поэтому приложения в контейнерах должны логировать в stdout, а не в файлы.

docker inspect — «полная информация о контейнере/образе»

docker inspect <ctr>                                    # полный JSON
docker inspect <ctr> --format '{{.State.Status}}'       # конкретное поле
docker inspect <ctr> --format '{{.State.ExitCode}}'     # код выхода
docker inspect <ctr> --format '{{.State.Pid}}'          # PID на хосте
docker inspect <ctr> --format '{{.HostConfig.Memory}}'  # лимит RAM
docker inspect <ctr> --format '{{.NetworkSettings.IPAddress}}'  # IP

inspect — это «рентген» для контейнера. Он показывает всё: конфигурацию, сетевые настройки, volumes, ресурсные ограничения, состояние.

docker events — «что происходит с Docker daemon прямо сейчас?»

docker events                           # все события в реальном времени
docker events --filter container=myapp   # события конкретного контейнера
docker events --since 10m               # за последние 10 минут
docker events --filter event=die         # только события гибели

events показывает поток событий: создание, запуск, остановку, гибель, перезапуск контейнеров. Незаменим при расследовании «почему контейнер рестартанул в 3 часа ночи».

docker stats — «сколько ресурсов потребляет?»

docker stats                    # все контейнеры, обновление в реальном времени
docker stats --no-stream        # один snapshot
docker stats <ctr>              # конкретный контейнер

Показывает: CPU%, Memory Usage/Limit, Network I/O, Block I/O, PIDs.

docker top — «какие процессы работают внутри?»

docker top <ctr>        # процессы с точки зрения хоста
docker top <ctr> aux    # расширенный формат (аналог ps aux)



Часть 3. Образы и Dockerfile

Длительность: ~45 минут

3.1 Анатомия Docker-образа: слои и CoW

Docker-образ — это не один файл и не одна файловая система. Это упорядоченная стопка read-only слоёв, каждый из которых представляет набор изменений (diff) файловой системы.

Как создаются слои

Каждая инструкция в Dockerfile, которая модифицирует файловую систему, создаёт новый слой:

FROM python:3.12-slim          # Слой 1: базовый образ (множество внутренних слоёв)
WORKDIR /app                   # Слой 2: создание директории
COPY requirements.txt ./       # Слой 3: один файл
RUN pip install -r requirements.txt  # Слой 4: установленные пакеты
COPY . .                       # Слой 5: исходный код

Инструкции ENV, EXPOSE, CMD, ENTRYPOINT, USER не создают файловых слоёв — они только модифицируют метаданные (конфигурацию) образа.

Дедупликация и sharing

Если на одном хосте запущено 50 контейнеров из образа python:3.12-slim, все 50 контейнеров разделяют read-only слои базового образа. На диске хранится только одна копия. Каждый контейнер имеет свой writable слой (container layer), который обычно минимален.

Это делает Docker невероятно эффективным по потреблению диска и по скорости запуска: не нужно копировать гигабайты файловой системы при каждом docker run.

Команды для исследования слоёв

docker history <image>                                    # слои и их размеры
docker inspect <image> --format '{{json .RootFS.Layers}}' # sha256 каждого слоя
docker image inspect <image> --format '{{.Size}}'         # общий размер

3.2 Инструкции Dockerfile: полный обзор

Dockerfile — это рецепт сборки образа. Каждая инструкция описывает один шаг.

FROM — базовый образ

FROM python:3.12-slim
FROM python:3.12-slim AS builder   # именованная стадия для multi-stage
FROM scratch                       # пустой образ (для Go-бинарников)

Всегда указывайте конкретную версию. FROM python:latest — антипаттерн: сборка будет давать разные результаты в разные дни.

WORKDIR — рабочая директория

WORKDIR /app    # создаёт директорию если не существует + переходит в неё

Предпочтительнее, чем RUN mkdir -p /app && cd /app, потому что WORKDIR действует на все последующие инструкции.

COPY vs ADD

COPY requirements.txt ./           # копирует файл из build context
COPY --from=builder /app/bin ./    # копирует из другой стадии multi-stage
ADD archive.tar.gz /opt/           # распаковывает tar-архивы автоматически
ADD https://example.com/file ./    # скачивает URL (плохая практика)

Правило: всегда используйте COPY, кроме случаев, когда вам явно нужна автоматическая распаковка tar-архива. ADD с URL — антипаттерн (не кешируется, не контролируется версия).

RUN — выполнение команд

# Exec form (предпочтительно)
RUN ["apt-get", "update"]

# Shell form
RUN apt-get update && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*

Каждый RUN создаёт слой. Объединяйте связанные команды через &&, чтобы не раздувать образ промежуточными слоями.

Очистка кеша пакетного менеджера (rm -rf /var/lib/apt/lists/*) в том же RUN обязательна — если сделать это в следующем RUN, файлы уже зафиксированы в предыдущем слое и не исчезнут.

ENV и ARG

ARG BUILD_DATE                  # доступна только при сборке (docker build)
ENV APP_PORT=8090               # доступна в runtime (внутри контейнера)

Ключевое отличие: ARG существует только на этапе docker build. ENV сохраняется в метаданных образа и доступна запущенному контейнеру.

Критически важно: ни ARG, ни ENV нельзя использовать для секретов. ARG виден в docker history, ENV виден через docker inspect и env внутри контейнера. Секреты передаются через runtime (volumes, Docker secrets, environment variables при запуске, внешние системы типа Vault).

EXPOSE — документация порта

EXPOSE 8090         # НЕ публикует порт на хост
EXPOSE 8090/tcp     # можно указать протокол

EXPOSE — это только метаданные, документация. Реальная публикация порта происходит при docker run -p 8090:8090.

USER — переключение пользователя

RUN addgroup --system app && adduser --system --ingroup app app
USER app            # все последующие инструкции и CMD выполняются от app

По умолчанию всё в Dockerfile выполняется от root. USER переключает пользователя для последующих RUN, CMD, ENTRYPOINT.

HEALTHCHECK — мониторинг изнутри

HEALTHCHECK --interval=15s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8090/healthz || exit 1

Docker периодически выполняет эту команду внутри контейнера. Если она возвращает ненулевой код выхода, контейнер получает статус unhealthy. Это используется для:





depends_on с condition: service_healthy в Compose



Автоматического restart (в связке с restart policy)



Мониторинга (видно в docker ps)

--start-period — grace period после запуска, пока приложение инициализируется.

CMD и ENTRYPOINT

# Только CMD — легко переопределить
CMD ["python", "app.py"]
# docker run myimage python other.py   ← переопределяет CMD полностью

# ENTRYPOINT + CMD — CMD становится аргументами
ENTRYPOINT ["python"]
CMD ["app.py"]
# docker run myimage other.py         ← подставляет 'other.py' как аргумент ENTRYPOINT

Рекомендация для production-сервисов:

ENTRYPOINT ["dumb-init", "--"]
CMD ["python", "app.py"]

3.3 Build cache: порядок инструкций = скорость сборки

Docker кеширует каждый слой. При повторной сборке, если инструкция и её контекст не изменились, Docker использует кешированный слой. Но: инвалидация кеша каскадная — если слой N изменился, все слои N+1, N+2, ... пересобираются.

# ✅ Правильно: зависимости (меняются редко) ПЕРЕД кодом (меняется часто)
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py ./

# ❌ Неправильно: любое изменение кода инвалидирует кеш pip install
COPY . .
RUN pip install --no-cache-dir -r requirements.txt

Принцип: располагайте инструкции от редко меняющихся (базовый образ, системные пакеты, зависимости) к часто меняющимся (исходный код). Это радикально ускоряет повторные сборки.

3.4 Multi-stage builds: разделение builder и runtime

Multi-stage build — одна из самых мощных техник оптимизации Docker-образов. Идея: использовать один или несколько промежуточных образов для сборки, а в финальный образ копировать только результат.

# Stage 1: Builder — содержит компилятор, зависимости для сборки
FROM golang:1.22 AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

# Stage 2: Runtime — минимальный образ с готовым бинарником
FROM gcr.io/distroless/static-debian12
COPY --from=builder /app /app
CMD ["/app"]

Результат: финальный образ содержит только бинарник — никакого компилятора, исходного кода, промежуточных файлов. Размер: 5–20 МБ вместо сотен мегабайт.

Преимущества:





Меньший размер образа → быстрее pull, меньше дискового пространства



Меньшая поверхность атаки → нет компилятора, нет shell, меньше CVE



Чистое разделение ответственности → builder и runtime — разные миры

3.5 Безопасность образа

Non-root пользователь

Большинство образов запускают процесс от root по умолчанию. Если атакующий выполнит container escape, он получит root на хосте. Решение:

RUN addgroup --system app && adduser --system --ingroup app app
USER app

.dockerignore

Замедляет сборку (Docker отправляет весь контекст демону)



Может привести к утечке секретов (.env попадает в слой)

# .dockerignore

__pycache__
*.pyc

Секреты в слоях

ARG виден в docker history --no-trunc. ENV виден через docker inspect. Ни то, ни другое нельзя использовать для паролей, токенов, ключей.

Правильные способы передачи секретов:





Docker Compose secrets (монтируются как файлы в /run/secrets/)

Выбор базового образа — одно из важнейших решений при создании Dockerfile. Он определяет размер, количество CVE, совместимость и удобство отладки.

Часть 4. Сети, хранение и оркестрация Compose

Длительность: ~30 минут

4.1 Сетевая модель Docker

Docker предоставляет несколько сетевых драйверов:

bridge (default) — виртуальный сетевой мост на хосте. Каждый контейнер получает IP-адрес в изолированной подсети. Контейнеры в одной bridge-сети видят друг друга. Для доступа с хоста нужна публикация порта (-p).

При создании пользовательской bridge-сети (docker network create mynet) контейнеры могут обращаться друг к другу по имени (Docker DNS). В дефолтной bridge-сети (docker0) DNS по именам контейнеров не работает — только --link (устаревший механизм).

host — контейнер использует сетевой стек хоста напрямую. Нет изоляции сети, нет NAT. Порт контейнера = порт хоста. Максимальная производительность сети, минимальная изоляция. Используйте осторожно.

none — контейнер полностью отключен от сети. Есть только loopback-интерфейс. Используется для максимальной изоляции.

overlay — для multi-host networking (Docker Swarm, Kubernetes). Позволяет контейнерам на разных физических машинах общаться как если бы они были в одной сети.

Под капотом bridge-сети

Когда Docker создаёт bridge-сеть:





Создаётся Linux bridge (виртуальный switch) на хосте



Для каждого контейнера создаётся пара veth (virtual ethernet) — один конец в bridge, другой в network namespace контейнера



Docker настраивает iptables-правила для NAT и port forwarding



При публикации порта (-p 8080:80) добавляется DNAT-правило

EXPOSE vs -p

EXPOSE 80 в Dockerfile — это только документация. Порт не публикуется на хост.

-p 8080:80 при docker run — реально создаёт iptables-правило для проброса порта 8080 хоста на порт 80 контейнера.

4.2 DNS и service discovery

В пользовательских Docker-сетях каждый контейнер автоматически получает DNS-запись. Docker встраивает DNS-сервер (127.0.0.11), который резолвит:





Имя контейнера → его IP



Network alias → IP контейнера (контейнер может иметь несколько alias)



Имя сервиса в Compose → IP всех контейнеров этого сервиса (round-robin)

# Внутри контейнера
nslookup api          # резолвит имя контейнера/сервиса в IP
nslookup db           # другой сервис в той же сети

Это позволяет контейнерам обращаться друг к другу по имени без знания IP-адресов. При пересоздании контейнера его IP может измениться, но имя остаётся.

4.3 Storage: volumes, bind mounts, tmpfs

По умолчанию все данные контейнера живут в writable layer и теряются при удалении контейнера. Для персистентности Docker предлагает три механизма:

Volumes (управляемые Docker)

docker volume create mydata
docker run -v mydata:/data myimage





Управляются Docker (хранятся в /var/lib/docker/volumes/)



Не зависят от структуры файловой системы хоста



Работают на всех платформах одинаково



Могут использовать драйверы для remote storage (NFS, AWS EBS и др.)



Лучший выбор для production-данных (БД, файловое хранилище)

Bind mounts (монтирование директории хоста)

docker run -v /host/path:/container/path myimage
docker run -v $(pwd)/src:/app/src myimage    # для dev: hot reload





Привязаны к конкретному пути на хосте



Полезны для разработки (монтирование исходного кода с hot reload)



Зависят от структуры ФС хоста



Потенциальный риск безопасности (контейнер видит файлы хоста)

tmpfs (RAM-диск)

docker run --tmpfs /tmp myimage





Данные в оперативной памяти



Исчезают при остановке контейнера



Используются для временных файлов, PID-файлов, кеша



Полезны с read_only: true (приложению нужен /tmp для работы)

4.4 Docker Compose: многоконтейнерные приложения

Docker Compose — инструмент для определения и запуска многоконтейнерных приложений через декларативный YAML-файл.

Структура compose.yaml

services:       # контейнеры (сервисы)
networks:       # пользовательские сети
volumes:        # именованные тома
configs:        # конфигурационные файлы
secrets:        # секреты (монтируются как файлы)

depends_on и service_healthy

Одна из важнейших конструкций в Compose:

# ❌ Плохо: api стартует, когда db ещё не готов принимать соединения
api:
  depends_on:
    - db

# ✅ Правильно: api ждёт, пока db не пройдёт healthcheck
api:
  depends_on:
    db:
      condition: service_healthy

depends_on без condition гарантирует только порядок запуска процессов, а не их готовность. Postgres-контейнер может запуститься за 200мс, но инициализация БД занимает 5 секунд — за это время API упадёт с «connection refused».

Profiles — опциональные сервисы

services:
  api:
    image: myapp:dev         # всегда запускается

  adminer:
    image: adminer:4
    profiles: [debug]        # только при --profile debug
    ports: ["8080:8080"]

  tests:
    profiles: [test]
    command: pytest

docker compose up -d                     # только обязательные
docker compose --profile debug up -d     # + debug-инструменты
docker compose --profile test run --rm tests  # запустить тесты

Override-файлы

compose.yaml               ← база (production)
compose.override.yaml      ← автозагружается для dev
compose.prod.yaml          ← явный override для prod

compose.override.yaml загружается автоматически при docker compose up. Для production:

docker compose -f compose.yaml -f compose.prod.yaml up -d

.env файл

Compose автоматически загружает .env из директории проекта:

POSTGRES_USER=appuser
POSTGRES_PASSWORD=secret
APP_PORT=8081

Проверка подстановки: docker compose config показывает итоговый конфиг с раскрытыми переменными.

4.5 Production patterns и переход к Kubernetes

Docker Compose отлично подходит для:





Локальной разработки



Интеграционного тестирования



Single-host deployments



Быстрого прототипирования

Но Compose не решает проблемы production-масштаба:





Self-healing на уровне кластера — Compose restart policy работает на одном хосте. Если хост упал — всё потеряно. Kubernetes перепланирует поды на другой узел.



Горизонтальное масштабирование — Compose может scale сервис, но без load balancer и без распределения по узлам.



Declarative rollout/rollback — Compose не поддерживает rolling update с canary, blue/green, или автоматический rollback при ошибках.



Scheduling — нет распределения контейнеров по узлам на основе ресурсов, affinity, taints/tolerations.

Security Hardening: 7 правил (подробный разбор)

Безопасность — не отдельный этап, а часть каждого решения. Каждый из этих пунктов — не рекомендация, а обязательный минимум для production.

Правило 1. Non-root пользователь

Большинство образов запускают процесс от root. Если атакующий выполнит container escape, он получит root на хосте. Это самая простая и критически важная мера.

# Debian-style
RUN addgroup --system app && adduser --system --ingroup app app
USER app

# Alpine-style
RUN addgroup -S app && adduser -S -G app app
USER app

Проверка:

docker inspect <container> --format '{{.Config.User}}'
# Должно быть: app (или любой uid != 0)

docker exec <container> id
# uid=100(app) gid=101(app) — НЕ root

Правило 2. Read-only корневая файловая система

В production приложение не должно писать в корневую ФС. Запись только в явно разрешённые директории через tmpfs.

services:
  app:
    read_only: true          # корневая ФС только для чтения
    tmpfs:
      - /tmp                 # RAM-диск для временных файлов
      - /var/run             # PID-файлы и сокеты

Проверка:

docker inspect <container> --format '{{.HostConfig.ReadonlyRootfs}}'
# true

Если приложение пишет логи в файл — перенаправьте в stdout. Если пишет в /tmp — это покрывается tmpfs. Всё остальное — volume для данных.

Правило 3. Drop all capabilities

Linux capabilities разбивают права root на ~40 отдельных привилегий. По умолчанию Docker оставляет ~14 capabilities. Принцип минимальных привилегий: сбросить всё, добавить только необходимое.

services:
  app:
    cap_drop:
      - ALL                  # сбрасываем все capabilities
    cap_add:
      - NET_BIND_SERVICE     # только если нужен порт < 1024

Частые capabilities и когда они действительно нужны:







Capability



Нужна для





NET_BIND_SERVICE



Биндинг на порты < 1024 (nginx на 443)





CHOWN



Смена владельца файлов в entrypoint





SETUID/SETGID



Su, sudo внутри контейнера (обычно не нужно)





SYS_PTRACE



strace, gdb — только для отладки!

Правило 4. no-new-privileges

Запрещает процессу повысить привилегии через setuid-биты или sudo. Без этого правила cap_drop можно обойти: если в образе есть бинарник с setuid-битом, процесс может восстановить сброшенные capabilities.

security_opt:
  - no-new-privileges:true

Это одна строка, которая закрывает целый класс атак. Нет причин не включать.

Правило 5. Секреты вне слоёв образа

Никогда не помещайте пароли, токены, ключи в Dockerfile — ни через ARG, ни через ENV. Они останутся в слоях образа навсегда.

Правило 6. Минимальный базовый образ

Чем больше пакетов в образе — тем больше CVE, больше attack surface. Выбирайте минимально достаточный base image:

ubuntu:22.04       →  ~80 МБ, много CVE, привычная среда
debian:bookworm-slim →  ~50 МБ, хороший баланс
python:3.12-slim   →  ~130 МБ, Python из коробки
python:3.12-alpine →  ~50 МБ, musl libc (осторожно с совместимостью!)
distroless         →  ~30 МБ, нет shell, нет pkg manager — идеал для prod
scratch            →  0 МБ, только для статических бинарников (Go, Rust)

Distroless — нет shell, нет менеджера пакетов, нет лишних инструментов. Отлично для production, сложнее для отладки. Для debug можно использовать distroless:debug (содержит busybox).

Правило 7. Сканирование CVE — Trivy

Trivy — сканер уязвимостей, секретов и мисконфигураций. Запуск без установки через Docker:

# Сканирование образа на CVE
docker run --rm aquasec/trivy:latest image myapp:dev

# Только HIGH и CRITICAL
docker run --rm aquasec/trivy:latest image \
  --severity HIGH,CRITICAL myapp:dev

# Сканирование Dockerfile на мисконфигурации
docker run --rm -v "$(pwd)":/work aquasec/trivy:latest config /work

# Сканирование на секреты в слоях
docker run --rm aquasec/trivy:latest image \
  --scanners secret myapp:dev

Что проверяет Trivy: CVE в OS-пакетах и зависимостях (pip, npm, go.sum), секреты в слоях (токены, ключи, пароли), мисконфигурации Dockerfile (root, latest, ADD вместо COPY).

В CI/CD: добавьте Trivy как обязательный шаг. HIGH + CRITICAL = блокировка релиза.

Registry и release flow

Теги: mutable vs immutable

Тег — это человекочитаемая ссылка на образ. Но тег mutable: можно перезаписать myapp:1.0.0 новым образом. Единственный immutable идентификатор — digest (sha256:...).

Правила release flow:





Используйте semver для тегов: 1.0.0, 1.1.0, 2.0.0



latest — антипаттерн для production. Не используйте его в deployment-манифестах.



В CI/CD деплойте по digest: myregistry/myapp@sha256:abc123...



Подписывайте образы (cosign/sigstore) для гарантии provenance
