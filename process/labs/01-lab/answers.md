## Задание 1. Файловые дескрипторы

### Что происходит при `open("/tmp/file.txt", O_RDWR)`

```text
Процесс (task_struct)        Ядро                          Диск
┌──────────────────┐    ┌──────────────────────┐    ┌──────────────┐
│ fd table:        │    │ Open File Table:     │    │ inode table: │
│ 0 → stdin        │    │ ┌──────────────────┐ │    │ ┌──────────┐ │
│ 1 → stdout       │    │ │ struct file:     │ │    │ │ inode 42 │ │
│ 2 → stderr       │    │ │  offset = 0     │─┼───→│ │ size     │ │
│ 3 ────────────────┼───→│ │  flags = O_RDWR │ │    │ │ blocks[] │ │
│                  │    │ │  f_inode → ─────┼─┘    │ │ refcount │ │
└──────────────────┘    │ └──────────────────┘ │    │ └──────────┘ │
                        └──────────────────────┘    └──────────────┘
```

1. Ядро находит наименьший свободный fd (3)
2. Создаёт `struct file` в системной таблице (с offset=0, flags=O_RDWR)
3. Связывает `struct file` с inode файла (через dentry lookup)
4. fd[3] в таблице процесса указывает на `struct file`

### Вопрос 1: fork() и общий файл

После `fork()` оба процесса разделяют один и тот же `struct file` → **общая позиция (offset)**.
Запись одного процесса двигает offset, и следующее чтение другого процесса начнётся с нового offset.
Это потому что `fork()` копирует таблицу fd, но не `struct file`.

### Вопрос 2: dup2 vs open

- `dup2(fd, 1)` — fd 1 теперь указывает на ту же `struct file`, что и fd. **Общий offset** и flags.
- Повторный `open()` — создаёт новый `struct file` с **отдельным offset** = 0. Независимая позиция.

## Задание 2. Жизненный цикл

```text
fork()
  │
  ▼
  R (Running/Runnable) ←───────────────────── SIGCONT
  │                                              ↑
  ├── scheduler → R (on CPU)                     │
  ├── I/O wait  → S (Interruptible Sleep)        │
  ├── disk I/O  → D (Uninterruptible Sleep)      │
  ├── SIGSTOP   → T (Stopped) ──────────────────┘
  │
  ▼
  exit() → Z (Zombie) → parent calls wait() → X (Dead, removed)
```

### Вопрос 1: D-state и SIGKILL

В D-state процесс ждёт завершения I/O на уровне ядра (например, NFS, disk). Ядро проверяет pending signals **только при возврате из kernel-mode в user-mode**. Пока процесс в D-state, он не возвращается → сигнал поставлен в очередь, но не доставлен.

### Вопрос 2: SIGSTOP vs SIGTSTP

| | SIGSTOP | SIGTSTP |
|---|---|---|
| Можно перехватить (trap)? | ❌ Нет | ✅ Да |
| Кто отправляет? | `kill -STOP` | Ctrl+Z (терминал) |
| Поведение по умолчанию | Остановить | Остановить |

### Вопрос 3: Зачем ждать wait()

Ядро хранит в зомби-записи:
- **Exit status** (код возврата) — нужен родителю
- **Resource usage** (rusage) — CPU time, memory peaks
- **Accounting data** — для системного учёта

Без `wait()` родитель не может узнать причину завершения потомка.

## Задание 3. fork, exec, clone

### Copy-on-Write

При `fork()` ядро НЕ копирует страницы памяти. Обе стороны получают read-only ссылки на те же страницы. Копирование происходит только при первой записи (page fault → copy page → mark writable). Для 10 ГБ процесса `fork()` копирует только page tables (~20 МБ), не данные.

### vfork()

Историческая причина: до COW fork() копировал всё → vfork() давал потомку адресное пространство родителя без копирования. Родитель блокировался до exec()/exit(). Сейчас полезен в embedded-системах с ограниченной памятью и без MMU.

### clone()

`clone()` — обобщённый fork(). Флаги контролируют что разделяется:

| Флаг | Что разделяет |
|------|--------------|
| `CLONE_VM` | Адресное пространство (память) |
| `CLONE_FILES` | Таблицу fd |
| `CLONE_FS` | Рабочий каталог, umask, root |
| `CLONE_SIGHAND` | Обработчики сигналов |

`pthread_create()` вызывает `clone(CLONE_VM | CLONE_FILES | CLONE_FS | CLONE_SIGHAND | ...)` → поток.

### exec()

`exec()` не создаёт процесс — заменяет код и данные в текущем PID:
- Старое адресное пространство удаляется
- Новый бинарник загружается
- PID, fd (без FD_CLOEXEC), сигналы (кроме ignored) сохраняются

### Пример fork() без exec()

- **Redis**: `fork()` для BGSAVE → потомок сериализует данные, родитель продолжает обрабатывать запросы. COW → потомок видит snapshot данных.
- **PostgreSQL**: `fork()` для каждого клиентского соединения.
- **Apache prefork**: pool fork'd workers ждёт запросов.

## Задание 6. Сигналы

### Вопрос 1: Доставка сигнала

Ядро проверяет pending signals при **переходе из kernel-mode в user-mode** (после syscall, interrupt, exception). Если сигнал не заблокирован — вызывается обработчик.

### Вопрос 2: Стандартные vs RT-сигналы

Стандартные (1–31): **один бит** в маске → повторный сигнал того же номера теряется.
Realtime (34–64): **очередь** (настоящий queue) → каждый сигнал доставляется, с данными (sigval).

### Вопрос 3: SIGTERM в D-state

Сигнал ставится в pending, но **не доставляется** до выхода из D-state. Если I/O никогда не завершится (мёртвый NFS) — процесс «завис навсегда».

### Вопрос 4: signal() vs sigaction()

| | `signal()` | `sigaction()` |
|---|---|---|
| Портабельность | Поведение зависит от ОС | Определено POSIX |
| Reset после вызова | Может сбросить на SIG_DFL | Нет (SA_RESTART) |
| Маска сигналов | Нет контроля | Полный контроль (sa_mask) |
| Рекомендация | Не использовать | Использовать всегда |

## Задание 7. inode и удалённые файлы

### Вопрос 1: hard link и unlink()

`rm` = `unlink()` → уменьшает link count на 1. Hard link — это запись в директории, указывающая на inode. Один inode может иметь несколько hard links.

### Вопрос 2: Когда inode освобождается

Два условия (оба должны быть true):
1. `link count == 0` (нет записей в директориях)
2. `open file count == 0` (ни один процесс не держит fd)

### Вопрос 3: df ≠ du на production

Типичный сценарий: logrotate удалил `/var/log/app.log`, но процесс не перечитал лог-файл → fd держит deleted inode → блоки заняты (df), но запись в директории отсутствует (du).

### Вопрос 4: Диагностика

```bash
sudo lsof +L1          # показывает файлы с link count < 1 (deleted)
# Найти процесс → перезапустить или отправить SIGHUP для reopen
```

## Задание 9. /proc

### Вопрос 1: Размер 0 байт

Файлы в /proc не существуют на диске. Контент генерируется ядром **при каждом read()** — поэтому size=0 (нечего считать заранее).

### Вопрос 2: Виртуальная FS

Данные хранятся в **структурах ядра в памяти** (task_struct, mm_struct и т.д.). При `cat /proc/PID/status` ядро читает эти структуры и форматирует в текст прямо в момент вызова.

### Context switches

| Тип | Когда | Преобладает у |
|-----|-------|---------------|
| voluntary | Процесс сам отдаёт CPU (sleep, I/O wait, mutex) | I/O-bound |
| nonvoluntary | Планировщик отбирает CPU (timeslice expired) | CPU-bound |

## Задание 12. Лимиты fd

### Почему fd может не совпасть с ulimit -n

`ulimit -n 256` → максимум 256 fd. Но fd 0, 1, 2 уже заняты + python сам открывает несколько fd → последний доступный ≈ 252–254.

### Порядок проверки

1. **Per-process** (ulimit/LimitNOFILE) — проверяется первым при `open()`
2. **System-wide** (fs.file-max) — проверяется если per-process пройден
3. **Per-user** (limits.conf) — устанавливается при login, становится per-process limit

## Задание 13. Diagnostic challenge — ответы

### Проблема 1: data-processor (fd leak)

**Симптомы:** `lsof -p PID | wc -l` → 200+ файлов
**Причина:** Открыто 200 fd на /dev/null, ни один не закрыт
**Решение:** Закрывать fd после использования (context manager / try-finally)

### Проблема 2: log-collector (zombie farm)

**Симптомы:** `ps aux | awk '$8 ~ /Z/'` → 3 зомби-процесса с PPID = log-collector
**Причина:** Родитель не вызывает wait() для потомков
**Решение:** Вызвать wait()/waitpid() или установить SIGCHLD → SIG_IGN

### Проблема 3: health-checker (CPU spin)

**Симптомы:** `top -p PID` → 100% CPU; `grep ctxt /proc/PID/status` → nonvoluntary >> voluntary
**Причина:** Бесконечный цикл без sleep/I/O
**Решение:** Добавить sleep или event-wait в цикл

### Проблема 4: cache-warmer (deleted file holding space)

**Симптомы:** `sudo lsof +L1` → PID держит deleted файл 50 МБ; `ls -la /proc/PID/fd/` → один fd → (deleted)
**Причина:** `unlink()` удалил запись из директории, но fd держит inode → блоки не освобождены
**Решение:** Перезапустить процесс или закрыть fd (через gdb: `call close(fd)`)

## Задание 14. strace

### Вопрос 1: syscall vs library call

- **Syscall** — переход в kernel-mode (int 0x80 / syscall instruction). Дорого (~100-1000 ns).
- **Library call** — обычный вызов функции в user-mode. Может внутри делать syscall, а может нет (strlen → нет, write → да).

### Вопрос 2: как strace работает

`strace` использует `ptrace(PTRACE_SYSCALL)` — ядро останавливает трассируемый процесс при каждом syscall entry и exit. strace читает регистры, декодирует аргументы, выводит.

### Вопрос 3: почему медленно

Каждый syscall → две остановки (entry + exit) → два переключения контекста → strace обрабатывает → продолжает. При 10000 syscall/s это 20000 дополнительных context switches.

## Задание 15. cgroups и namespaces

### Вопрос 1: PID namespace и ps

Внутри PID namespace процесс видит только процессы в этом namespace. PID 1 внутри = bash, PID 2 = ps. Остальные сотни процессов хоста невидимы.

### Вопрос 2: PID 1 внутри namespace умирает

Ядро посылает SIGKILL **всем** процессам в этом PID namespace. Namespace уничтожается. Это аналогично «остановке контейнера».

### Вопрос 3: cgroups + namespaces = контейнер

- **namespaces** → изоляция видимости (процессы, сеть, файловая система)
- **cgroups** → ограничение ресурсов (CPU, RAM, I/O)
- **Вместе** → процесс думает что он один в системе, и не может потребить больше выделенного = контейнер
