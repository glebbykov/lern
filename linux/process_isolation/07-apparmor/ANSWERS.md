# 07 — AppArmor: контрольные вопросы

**В чём разница AppArmor vs SELinux?**
| | AppArmor | SELinux |
|---|---|---|
| Привязка | путь исполняемого файла | label на inode |
| Сложность | низкая, ~Apache config | высокая, MLS/MCS/RBAC |
| Дистрибутивы | Ubuntu, Debian, SUSE | RHEL, Fedora, CentOS |
| Учить | за день | за месяц |
| Дебаг | `dmesg | grep DENIED` + `aa-logprof` | `ausearch` + `audit2allow` |

**Что значит «MAC» и почему он сильнее DAC?**
DAC — *владелец* решает, кто читает его файлы (chmod 600). MAC —
*администратор* через политику, программа изменить не может, даже
из-под root. Защищает не от пользователя, а от *скомпрометированного
процесса с правами root*.

**Что делать если AppArmor сломал работающее приложение?**
1. `aa-complain /path/to/binary` — переводим в complain-mode, нарушения
   только логируются.
2. Воспроизводим проблему, собираем `dmesg | grep DENIED`.
3. `aa-logprof` — интерактивный инструмент, добавит правила в профиль.
4. `aa-enforce` — обратно в боевой режим.

**Защитит ли AppArmor от модификации профиля изнутри контейнера?**
Нет, если внутри контейнера есть `CAP_MAC_ADMIN` (или root + доступ
к `/etc/apparmor.d/`). Поэтому: профили грузятся **на хосте**,
изнутри контейнера их менять нельзя; в Docker `CAP_MAC_ADMIN` не
выдаётся по умолчанию.

**Почему `#include <abstractions/base>`?**
В `/etc/apparmor.d/abstractions/` лежат типовые блоки разрешений:
`base` (доступ к /dev/null, libc, locale), `bash`, `nameservice` и т.д.
Без `base` придётся вручную перечислять десятки путей, которые любой
бинарь читает на старте.
