# Лабораторная работа 2: Продвинутая изоляция процессов в Linux

## Цель работы
Изучить механизмы управления ресурсами (Control Groups v2), а также продвинутые пространства имен (Mount, User). Научиться собирать полноценный легковесный контейнер без использования Docker.

## Теоретические сведения
В первой части мы изучили `chroot`, `PID`, `UTS` и `Network` namespaces. В этой части мы рассмотрим:
1.  **Cgroups (Control Groups)** — механизм ядра для ограничения, контроля и учета потребления ресурсов (CPU, память, диск I/O) группами процессов. Современные дистрибутивы (в т.ч. Ubuntu 22.04) используют единую иерархию **Cgroups v2**.
2.  **Mount Namespace** — изолирует точки монтирования. Позволяет процессу иметь собственную файловую систему (например, свой `/tmp`), не затрагивая основную ОС.
3.  **User Namespace** — позволяет процессу иметь привилегии `root` внутри пространства имен, оставаясь обычным (unprivileged) пользователем снаружи. Это основа "Rootless" контейнеров.

## Часть 1: Ограничение ресурсов (Cgroups v2)

В Cgroups v2 управление происходит через виртуальную файловую систему, примонтированную в `/sys/fs/cgroup`.

### Практическое задание 1: Ограничение памяти (RAM)
1. Создайте новую контрольную группу (директорию):
   ```bash
   mkdir -p /sys/fs/cgroup/lab_mem
   ```
2. Установите жесткий лимит памяти в 50 Мегабайт:
   ```bash
   echo "50M" > /sys/fs/cgroup/lab_mem/memory.max
   ```
3. Запустите скрипт из песочницы (он будет бесконечно выделять память), предварительно поместив текущую оболочку в группу:
   ```bash
   # Открываем новую оболочку, помещаем ее в cgroup
   echo $$ > /sys/fs/cgroup/lab_mem/cgroup.procs
   # Запускаем утечку памяти
   /root/sandboxes/mem_leak.py
   ```
   *Заметьте: процесс будет убит (Killed), а в выводе `dmesg | tail` вы увидите срабатывание OOM-Killer именно для этой cgroup.*

### Практическое задание 2: Ограничение процессора (CPU)
1. Создайте группу для процессора:
   ```bash
   mkdir -p /sys/fs/cgroup/lab_cpu
   ```
2. Ограничьте потребление до 10% одного ядра. Формат `cpu.max` — это `<КВОТА> <ПЕРИОД>`.
   ```bash
   echo "10000 100000" > /sys/fs/cgroup/lab_cpu/cpu.max
   ```
3. Запустите бесконечный цикл в фоне и поместите его PID в группу:
   ```bash
   while true; do :; done &
   PID=$!
   echo $PID > /sys/fs/cgroup/lab_cpu/cgroup.procs
   ```
4. Проверьте потребление с помощью `top -p $PID`. Оно не превысит 10.0%. После проверки убейте процесс (`kill $PID`).

## Часть 2: Изоляция точек монтирования (Mount Namespace)

### Практическое задание 3
1. Запустите изолированную оболочку с приватными точками монтирования:
   ```bash
   unshare -m bash
   mount --make-rprivate /  # Отвязываем от хостовой системы
   ```
2. Смонтируйте временную файловую систему в `/mnt`:
   ```bash
   mount -t tmpfs none /mnt
   touch /mnt/secret_file
   ls /mnt
   ```
3. Откройте второй терминал (на хосте) и выполните `ls /mnt`. Вы не должны увидеть `secret_file`.

## Часть 3: Rootless контейнер (User Namespace)

### Практическое задание 4
1. Создайте обычного пользователя и переключитесь на него:
   ```bash
   useradd -m labuser
   su - labuser
   ```
2. Запустите User Namespace с маппингом прав пользователя на root:
   ```bash
   unshare -U -r bash
   ```
3. Проверьте свои права внутри:
   ```bash
   id  # Вы увидите uid=0(root)
   touch /tmp/my_file
   ```
4. Вернитесь в терминал хоста (от обычного пользователя root) и проверьте владельца файла:
   ```bash
   ls -n /tmp/my_file
   ```
   *Заметьте: снаружи владельцем файла является пользователь `labuser` (UID 1001), хотя внутри он создавался от имени `root`.*

## Самостоятельная работа: Сборка "Контейнера"
Используя знания из двух лабораторных работ, напишите bash-скрипт `run_container.sh`, который:
1. Создает cgroup с лимитом CPU (20%) и RAM (100M).
2. Подготавливает новую "тюрьму" (rootfs) с помощью `busybox`:
   ```bash
   mkdir -p /tmp/container_root/{bin,proc}
   # Копируем статически слинкованный busybox (содержит все утилиты: ls, ps, sh, ip)
   cp /bin/busybox /tmp/container_root/bin/
   # Создаем симлинки для базовых команд ВНУТРИ будущей тюрьмы
   chroot /tmp/container_root /bin/busybox --install -s /bin
   ```
3. Запускает команду `unshare`, изолируя **сразу 4 пространства**: PID, UTS, Mount и Cgroup.
4. Меняет корень (`chroot` в `/tmp/container_root`) и монтирует `/proc`.
5. Выполняет внутри команды `hostname container`, `id` и `ps aux` для проверки полной изоляции.

## Дополнение: Продвинутая изоляция (blkio и pivot_root)

### 1. Ограничение дискового ввода-вывода (blkio / io)
В Cgroups v1 контроллер назывался `blkio`. В Cgroups v2 он называется `io`. Он позволяет ограничить скорость чтения/записи (IOPS и BPS) для конкретного блочного устройства.

**Практика:**
```bash
# 1. Узнаем major:minor номер нашего диска (например, /dev/sda или /dev/root)
lsblk -o NAME,MAJ:MIN
# 2. Создаем группу и ставим лимит на запись (например, 1 МБ/с) для устройства 8:1 (замените на свои)
mkdir -p /sys/fs/cgroup/lab_io
echo "8:1 wbps=1048576" > /sys/fs/cgroup/lab_io/io.max
# 3. Тестируем запись через dd внутри группы.
```

### 2. Pivot_root вместо Chroot
`chroot` уязвим к выходу из "тюрьмы", так как он не меняет саму структуру монтирования. В настоящих контейнерах (Docker, LXC) используется системный вызов `pivot_root`. Он берет текущий корень (`/`) и переносит его в указанную папку, а новую папку делает настоящим корнем.

**Практика:**
```bash
# pivot_root требует, чтобы старый и новый корень были на разных файловых системах.
unshare -m bash
mount -t tmpfs tmpfs /tmp/new_root
mkdir -p /tmp/new_root/put_old_here
# Копируем минимальное окружение
cp -r /bin /lib /lib64 /tmp/new_root/
# Делаем pivot_root
cd /tmp/new_root
pivot_root . put_old_here
# Отмонтируем старую систему
umount -l /put_old_here
```
