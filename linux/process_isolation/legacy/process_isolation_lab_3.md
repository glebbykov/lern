# Лабораторная работа 3: Продвинутая безопасность и фильтрация системных вызовов

## Цель работы
Изучить механизмы ограничения полномочий процессов на уровне ядра ОС Linux: Capabilities, Seccomp и Mandatory Access Control (AppArmor). Понять, как современные контейнеры (Docker/Kubernetes) обеспечивают безопасность приложений.

## Теоретические сведения
Стандартная модель безопасности Linux делит пользователей на привилегированных (`root`, UID 0) и непривилегированных. Если процессу нужно открыть порт ниже 1024 (например, 80 для веб-сервера), он должен работать от `root`, что небезопасно. 

Для решения этих проблем существуют механизмы:
1.  **Linux Capabilities** — разбивают права `root` на десятки мелких привилегий. Можно дать программе право открывать 80 порт, но запретить изменять файлы системы.
2.  **Seccomp (Secure Computing)** — фильтрация системных вызовов к ядру. Если приложение попытается выполнить запрещенный вызов (например, форматирование диска `mount`), ядро немедленно убьет процесс (SIGSYS).
3.  **AppArmor (Mandatory Access Control)** — механизм мандатного контроля доступа. Позволяет администратору создавать профили, жестко описывающие, к каким файлам процесс имеет доступ, даже если процесс работает от `root`.

## Часть 1: Linux Capabilities (Дробление прав Root)

В этом задании мы заставим веб-сервер открывать привилегированный 80-й порт от имени самого бесправного пользователя в системе — `nobody`.

### Практическое задание 1
1. Создайте копию интерпретатора Python, чтобы не менять системный бинарник:
   ```bash
   cp /usr/bin/python3 /tmp/pyweb
   ```
2. Попробуйте запустить веб-сервер на 80 порту от имени пользователя `nobody` (вы получите ошибку *Permission denied*):
   ```bash
   su -s /bin/bash nobody -c "/tmp/pyweb -m http.server 80"
   ```
3. Выдайте копии Python привилегию (Capability) `CAP_NET_BIND_SERVICE` (право открывать порты < 1024):
   ```bash
   setcap cap_net_bind_service+ep /tmp/pyweb
   ```
4. Проверьте, что атрибут установлен:
   ```bash
   getcap /tmp/pyweb
   ```
5. Снова запустите сервер:
   ```bash
   su -s /bin/bash nobody -c "/tmp/pyweb -m http.server 80 &"
   PID=$!
   sleep 1
   curl -I localhost:80  # Сервер успешно ответит!
   kill $PID
   ```
   *Вывод: процесс успешно открыл порт без прав `root`.*

## Часть 2: Seccomp (Блокировка системных вызовов)

Для демонстрации мы используем интеграцию `seccomp` в `systemd`, которая позволяет ограничивать вызовы без написания C-кода.

### Практическое задание 2
1. Выполните команду `uname`, которая запрашивает у ядра информацию об ОС:
   ```bash
   uname -a
   ```
2. Теперь запустим ту же команду через `systemd-run`, но запретим ей делать системный вызов `uname` (аргумент `SystemCallFilter=~uname`, тильда означает запрет):
   ```bash
   systemd-run --wait -p SystemCallFilter=~uname uname -a
   ```
3. Проверьте вывод.
   *Заметьте: процесс мгновенно завершится с ошибкой (status=SYS). Ядро (через Seccomp) убило программу, как только она попыталась запросить запрещенную информацию.*

## Часть 3: AppArmor (Мандатный контроль)

Создадим скрипт, которому запрещено читать системный файл с пользователями `/etc/passwd`.

### Практическое задание 3
1. Напишите тестовый скрипт `/tmp/test_aa.sh`:
   ```bash
   cat << 'EOF' > /tmp/test_aa.sh
   #!/bin/bash
   cat /etc/passwd > /dev/null && echo "УСПЕХ: Файл прочитан!" || echo "ОТКАЗ: Доступ запрещен!"
   EOF
   chmod +x /tmp/test_aa.sh
   ```
2. Запустите скрипт без защиты:
   ```bash
   /tmp/test_aa.sh  # Должен вывести "УСПЕХ"
   ```
3. Создайте профиль AppArmor для этого скрипта (сохраните в `/etc/apparmor.d/tmp.test_aa.sh`):
   ```text
   #include <tunables/global>

   /tmp/test_aa.sh {
     #include <abstractions/base>
     #include <abstractions/bash>
     /usr/bin/bash ix,
     /usr/bin/cat ix,
     /tmp/test_aa.sh r,       # Право скрипта прочитать самого себя
     deny /etc/passwd r,      # ЖЕСТКИЙ ЗАПРЕТ на чтение passwd
     /dev/null rw,
   }
   ```
4. Загрузите профиль в ядро и запустите скрипт снова:
   ```bash
   apparmor_parser -r /etc/apparmor.d/tmp.test_aa.sh
   /tmp/test_aa.sh
   ```
   *Заметьте: несмотря на то, что скрипт запускается от имени `root`, AppArmor заблокирует чтение файла (`cat: /etc/passwd: Permission denied`).*

5. Очистите за собой, выгрузив профиль из ядра и удалив файл:
   ```bash
   apparmor_parser -R /etc/apparmor.d/tmp.test_aa.sh
   rm /etc/apparmor.d/tmp.test_aa.sh
   ```

## Дополнение: Seccomp-BPF и MAC-системы (AppArmor vs SELinux)

### 1. Seccomp-BPF
Обычный Strict Seccomp разрешает только 4 системных вызова: `read`, `write`, `_exit`, `sigreturn`. Это слишком жестко.
**Seccomp-BPF (Berkeley Packet Filter)** позволяет писать сложные фильтры, разрешая одни сисколлы (например, `open`) и блокируя другие (например, `reboot`), а также фильтровать по аргументам сисколла. Docker использует именно Seccomp-BPF по умолчанию, блокируя ~44 опасных сисколла из 300+.

### 2. Mandatory Access Control (AppArmor и SELinux)
В Linux существуют две основные системы мандатного контроля доступа:
*   **AppArmor:** Используется по умолчанию в Ubuntu/Debian/SUSE. Привязывает профили безопасности к путям исполняемых файлов. Управление проще.
*   **SELinux (Security-Enhanced Linux):** Разработка АНБ США, стандарт для RHEL/CentOS/Fedora. Использует метки (контексты) файлов, процессов и портов (например, `system_u:object_r:httpd_sys_content_t:s0`). Считается более сложной, но более гибкой.

**Практика:**
```bash
# Проверка статуса AppArmor (покажет загруженные профили и режим Enforce/Complain)
aa-status

# Проверка статуса SELinux (если он не установлен, команда вернет ошибку или покажет Disabled)
getenforce || echo "SELinux не установлен"
```
