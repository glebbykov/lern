# 08 — OverlayFS: контрольные вопросы

**Какую проблему решает OverlayFS vs `cp -r`?**
- Дисковое место: 100 контейнеров из одного image занимают 1 image
  + 100 маленьких diff'ов, не 100 копий.
- Скорость старта: `mount` — миллисекунды; `cp -r` — секунды/минуты.
- Read-only слой защищён от случайных правок «другим контейнером».

**Что такое whiteout физически?**
Character device с major=0, minor=0. При `readdir` ядро встречает
такой файл в upper и не выдаёт одноимённый из lower. На реальной ФС
это просто `mknod` с этими номерами — никаких extended attributes.

**Зачем workdir?**
Для атомарных операций ядра. Когда CoW копирует файл из lower в upper,
он сначала создаёт его в workdir, потом переименовывает в upper —
иначе при сбое останутся «полусоздавшиеся» файлы. workdir должен быть
**на той же ФС**, что и upper.

**Как Docker строит image из Dockerfile?**
Каждая инструкция (`RUN`, `COPY`, `ADD`) создаёт новый слой = новый
upperdir. После окончания шага этот upper становится lower для
следующей инструкции. К моменту `docker push` все слои упакованы
в tarballs и загружаются в registry.

**Почему `RUN apt install x && apt clean` в одной строке экономит место?**
Каждая `RUN` — отдельный слой. Если в одном слое поставить пакет,
а в следующем — почистить кеш, то **оба состояния** остаются в image
(удалённые файлы превращаются в whiteout, но reверхний слой содержит
их копии). Объединение в одну `RUN` уменьшает финальный размер.
