# 02 — Namespaces: контрольные вопросы

**Что произойдёт с процессами в PID-ns, если PID 1 завершится?**
Ядро отправит `SIGKILL` всем оставшимся процессам в этом namespace.
PID 1 — init для namespace, без него остальные не могут существовать.
В Docker это причина, по которой контейнер «умирает» вместе с процессом
из CMD/ENTRYPOINT.

**Какие 8 типов namespaces бывают?**
UTS, PID, MNT, NET, IPC, USER, CGROUP, TIME (с ядра 5.6). Разработка
ещё одного — `clock` — обсуждается в LKML.

**Как проверить, в каких namespaces процесс?**
`ls -lL /proc/<pid>/ns/` — каждая запись это symlink с inode-номером
namespace. Два процесса с одинаковым inode для UTS — в одном UTS-ns.
Так Docker определяет «принадлежит контейнеру».

**Зачем USER namespace для rootless?**
Внутри user-ns можно быть uid 0 (root), снаружи остаться uid 1000.
Все capabilities, выданные внутри, действуют ТОЛЬКО на ресурсы
этого namespace — на хост повлиять нельзя. Базис для rootless Docker
и Podman.

**Почему `unshare --pid` без `--fork` не даёт PID 1 = 1?**
`unshare(CLONE_NEWPID)` создаёт новый PID-ns, но текущий процесс
остаётся в старом — у него уже есть PID. PID 1 получит только первый
*потомок* нового процесса. Поэтому `--fork` обязателен.
