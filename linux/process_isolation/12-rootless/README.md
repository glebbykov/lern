# Этап 12: Rootless Контейнеры (User Namespaces)

В современном мире запускать контейнеры от имени пользователя `root` (UID 0 на хосте) считается небезопасным (антипаттерн). 
Даже если процесс изолирован, в случае уязвимости ядра злоумышленник, вырвавшийся наружу, сохранит права суперпользователя.

Чтобы этого избежать, используются "Rootless" (бесправные) контейнеры. Их суть в том, что снаружи (на хосте) контейнер запускается от имени обычного пользователя (например, UID 1000), а внутри своего пространства он мапится в UID 0 (root).

## Теория и команды

### 1. Как работает маппинг?
За маппинг отвечают файлы `/etc/subuid` и `/etc/subgid`. В них прописывается, какие "вложенные" UID может использовать пользователь.
Пример строки в `/etc/subuid`:
`ubuntu:100000:65536`
Это значит, что пользователь `ubuntu` может отображать UID внутри контейнера на UID от 100000 до 165535 на хосте.

### 2. Практика: unshare с маппингом
Для создания User Namespace мы используем флаг `-U` в команде `unshare`.

```bash
# Создаем пользователя для тестов
sudo useradd -m labuser

# Переключаемся на пользователя
su - labuser

# Изолируем User и Mount пространства, и маппим текущего пользователя в root (-r)
unshare -U -m -r bash

# Проверяем кто мы ВНУТРИ
id
# Вывод: uid=0(root) gid=0(root)

# Создаем файл
touch /tmp/rootless-test
```

Если теперь открыть второй терминал на хосте и посмотреть владельца файла `ls -l /tmp/rootless-test`, мы увидим, что им владеет `labuser`, а не `root`.
Таким образом, процесс имеет полные права *внутри* своей песочницы, но *снаружи* он абсолютно безопасен.

В реальных движках (Podman, Docker Rootless) для сложного маппинга используются утилиты `newuidmap` и `newgidmap`.
