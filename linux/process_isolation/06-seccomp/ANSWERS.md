# 06 — Seccomp: контрольные вопросы

**Какой сигнал получает процесс при запрещённом syscall?**
`SIGSYS` (Bad system call), сигнал 31. По умолчанию вызывает abort
с core dump. Можно переопределить ответ фильтра через
`SECCOMP_RET_ERRNO`, тогда вместо kill вернётся `errno` и процесс
просто получит ошибку.

**Что означает `PR_SET_NO_NEW_PRIVS`?**
Гарантия ядра: процесс и его потомки никогда не смогут получить новые
привилегии (через setuid/file caps/LSM transitions). Это
**обязательное предусловие** для seccomp без CAP_SYS_ADMIN — иначе
непривилегированный пользователь мог бы поставить хитрый фильтр на
SUID-программу.

**Чем seccomp-bpf отличается от strict?**
Strict разрешает только 4 syscalls (read/write/_exit/sigreturn) —
бесполезно для реальных программ. BPF — полноценная программа,
решающая для каждого syscall (видит nr и аргументы). Может допускать
одни, блокировать другие, фильтровать по аргументам.

**Можно ли фильтровать по аргументам syscall?**
Да, BPF-программа получает структуру `seccomp_data` с nr и 6 аргументами.
Но **указатели на пользовательскую память читать нельзя** — TOCTOU
атака. Поэтому фильтровать `open("path")` по строке невозможно,
только по флагам (`O_WRONLY`, `O_CREAT` — числа).

**Где взять номер syscall для своей архитектуры?**
- Linux: `ausyscall x86_64 uname` или `cat /usr/include/asm/unistd_64.h`
- Шпаргалка: https://chromium.googlesource.com/chromiumos/docs/+/HEAD/constants/syscalls.md

Номера разные для x86_64 и arm64 — фильтр должен это учитывать
(сначала проверь `seccomp_data.arch`).
