# 09 — Сеть контейнеров: veth, bridge, NAT

## Идея

Сетевой namespace (этап 02) даёт пустой стек. Чтобы из него ходить
наружу, нужны:

1. **veth pair** — виртуальный «патч-корд», два конца. Что вошло
   в один — мгновенно вышло из другого. Один конец живёт в host
   netns, второй — в контейнере.
2. **bridge** — программный коммутатор уровня L2 (`brctl` / `ip link
   add type bridge`). К нему подключаются veth-концы от хоста — все
   контейнеры оказываются в одном L2-домене.
3. **NAT** — `iptables -t nat -A POSTROUTING -j MASQUERADE` на исходящем
   хост-интерфейсе. Контейнеры получают приватные IP, но во внешний
   мир ходят с IP хоста.

Это ровно то, что Docker делает по умолчанию (сеть `bridge`, мост
`docker0`, дефолтная подсеть `172.17.0.0/16`).

## Что делаем

1. **Часть 1 — veth pair**: два netns соединяем напрямую, ping друг
   друга работает, но в интернет — нет.
2. **Часть 2 — bridge**: ставим мост, подключаем 3 контейнера, между
   собой пингуются.
3. **Часть 3 — NAT outbound**: даём мосту IP, делаем default route в
   контейнерах, включаем `net.ipv4.ip_forward=1` и MASQUERADE на
   исходящем интерфейсе хоста — теперь из контейнеров ping
   `1.1.1.1` идёт.

## Запуск

```bash
sudo ./run.sh    # все 3 части подряд, в конце чистится
sudo ./check.sh  # автотест ping между ns + наружу
```

## Карта в Docker

| Здесь | Docker |
|---|---|
| `ip link add docker0 type bridge` | автоматически при первом контейнере |
| `ip link add vethXX type veth peer ...` | для каждого контейнера |
| `iptables ... -j MASQUERADE` | правило в Docker chain |
| `ip netns exec` | `nsenter --net=/proc/<pid>/ns/net` |
