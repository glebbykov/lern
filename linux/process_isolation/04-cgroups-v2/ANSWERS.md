# 04 — cgroups v2: контрольные вопросы

**В чём принципиальное отличие v2 от v1?**
v1 — параллельные иерархии для каждого контроллера (cpu, memory, blkio
жили в разных папках). Процесс мог быть в *разных* группах для разных
контроллеров — путаница. v2 — единая иерархия, процесс ровно в одной
группе, контроллеры включаются/отключаются на уровне группы через
`cgroup.subtree_control`.

**Что произойдёт при превышении memory.max?**
Сначала ядро попытается отдать memory back: page cache, swap (если
не запрещён через `memory.swap.max=0`). Если не помогло — OOM-killer,
либо процесс, либо самый большой в группе. Счётчик в `memory.events`.

**Что такое PSI и зачем она?**
Pressure Stall Information — `*.pressure` файлы. Показывают долю
времени, когда хотя бы один процесс группы блокировался на ресурсе.
В отличие от старых `memory.usage_in_bytes` это интегральная метрика
*страдания*, удобна для autoscaler-а.

**Чем `cpu.max` отличается от `cpu.weight`?**
`cpu.max` — жёсткий лимит (квота/период, мкс). `cpu.weight` (1-10000) —
относительный share, действует только при контеншене. Docker по
умолчанию ставит `--cpu-shares` (weight), `--cpus=N` это `cpu.max`.

**Почему нужно `+memory +cpu` в `cgroup.subtree_control` родителя?**
Контроллеры в v2 включаются «сверху вниз». Чтобы дочерняя группа
могла иметь `memory.max`, родитель должен делегировать ей контроллер
memory через subtree_control. Это «no internal processes» правило:
группа либо имеет процессы, либо подгруппы — не одновременно.
