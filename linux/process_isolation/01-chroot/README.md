# 01 — chroot: первая попытка изоляции

## Идея

`chroot(2)` меняет видимый корневой каталог процесса. Самый старый
механизм изоляции в Unix (1979 год). Изолирует **только файловую
систему** — всё остальное (PID, сеть, hostname, IPC, пользователи)
остаётся общим с хостом.

В контейнере Docker эту работу делает распаковка слоёв image в
`/var/lib/docker/overlay2/.../merged` + переход рантайма в этот корень.

## Что делаем

1. Собираем минимальный rootfs из статического `busybox` (один
   бинарник + симлинки на все утилиты, никаких `.so`).
2. Заходим внутрь через `chroot`.
3. Демонстрируем: ФС изолирована (видим свой `/etc/hostname`), всё
   остальное — нет (`hostname` хоста, `ps` хоста, сеть хоста).
4. **Сбегаем из chroot** через `/proc/1/root` — этого классическая
   `chroot` не предотвращает.

## Запуск

```bash
sudo ./run.sh     # пошаговый показ
sudo ./check.sh   # автотест — собирает, заходит, выходит, чистит
```

## Что считается «правильно прошёл»

- Файл, созданный внутри chroot, лежит в каталоге rootfs хоста.
- `hostname` внутри chroot тот же, что снаружи (UTS не изолирован).
- Команда `chroot /proc/1/root /bin/sh` *успешно* запускает шелл в
  корне хоста — это и есть тот самый «классический побег».
  Защита от него — следующий этап (`02-namespaces` + `03-pivot-root`).

## Карта в Docker-термины

| Делаем здесь | Эквивалент в Docker |
|---|---|
| `mkdir -p $ROOT/{bin,...}` | распаковка слоя image |
| `cp /bin/busybox $ROOT/bin/` | `FROM scratch` + `ADD busybox` |
| `chroot $ROOT /bin/sh` | старт процесса в `merged/` |
| `mount --rbind /dev $ROOT/dev` | `--mount type=bind,source=/dev` (не делается!) |

Docker не использует *чистый* chroot — он использует `pivot_root` (этап 03),
именно потому что побег через `/proc/1/root` слишком тривиален.
